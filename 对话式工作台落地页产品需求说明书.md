# 对话式工作台落地页产品需求说明书

> 版本：v1.1 | 日期：2026-07-11
> 定位：定义客户首访订单广场时"分类引导 + AI 对话 + 订单预览"的三栏式工作台落地页设计，替代当前卡片式 CustomerWorkbench。
> 关联需求：客户发布订单与审核 PRD §2.1（AI对话创建）、订单详情页弱结构化展示设计说明、CSDN订单管理系统整体方案 §6.3

---

## 1. 问题陈述

### 1.1 现状

客户（发单方）首访订单广场时的落地页当前为**卡片式工作台**（CustomerWorkbench.tsx）——统计卡片 + 快捷发单入口 + 订单列表 + 近期动态。客户要发单需点击"AI智能发单"跳转到独立的 OrderPublishPage 表单页完成填写。

### 1.2 问题

1. **发单入口过深**：客户落地后看到的是统计数据和订单列表，而非"开始描述你的需求"。发单动作需要点击→跳转→填表单，路径偏长。
2. **AI对话发单与落地页割裂**：PRD §2.1 已定义"AI对话创建"为核心发单模式，但当前落地页没有体现对话主体地位，AI对话藏在二级页面里。
3. **工作台信息密度高但行动力低**：统计卡片、近期动态等模块是"展示型"信息，客户落地后的核心意图（发单 / 查看订单进度）没有被第一优先级响应。
4. **甲方明确期望对话模式**：甲方希望落地页以对话框为主体，客户可以直接在对话中完成发单，类似 WorkBuddy / ChatGPT 的交互范式。

### 1.3 不解决的问题

- 接单方（user 角色）的落地页本期不调整，继续使用现有订单广场；本次分类引导仅用于发单方新建订单。
- 运营端的页面不在本次范围。
- 已提交订单的详情页、里程碑、账单等流程不在本次范围。

---

## 2. 目标

### 2.1 用户目标

1. **降低发单启动门槛**：客户落地后无需跳转，直接在对话输入框描述需求即可启动发单流程。
2. **对话即草稿，所见即所得**：客户在对话中逐步完善需求的同时，右侧实时看到订单草稿成型，降低"填表单"的心理负担。
3. **历史对话与历史订单统一管理**：客户在同一个界面内管理"未完成的草稿"和"已提交的订单"，无需在多个页面间跳转。

### 2.2 业务目标

1. **提升发单转化率**：缩短发单路径（从"落地→跳转→填表单"变为"落地→对话"），预期发单完成率提升 20%+。
2. **强化 AI 驱动定位**：对话式落地页直接体现平台的 AI 驱动特性，与纯表单型竞品形成差异化。
3. **降低客户学习成本**：对话模式符合用户对 AI 产品的直觉认知，新客户无需学习"哪个字段填什么"。

### 2.3 成功指标

| 指标类型 | 指标名称 | 基线（当前） | 目标 | 测量方式 |
|---------|---------|-------------|------|---------|
| 领先 | 落地页→发单启动率 | 未埋点 | ≥60%（落地后30秒内开始对话） | 埋点：首次输入框输入 |
| 领先 | 对话→草稿生成率 | 未埋点 | ≥45%（开始对话的用户中生成草稿的比例） | 埋点：右侧预览区首次出现 |
| 领先 | 草稿→提交审核率 | 未埋点 | ≥70%（生成草稿后提交审核的比例） | 埋点：提交按钮点击 |
| 滞后 | 发单整体完成率 | 现有 OrderPublishPage 转化率 | 对比提升 ≥20% | A/B 测试 |
| 健康 | 对话平均轮次 | — | 3-8轮（过低=信息不足，过高=AI追问效率低） | 埋点：对话轮次统计 |

---

## 3. 非目标

1. **不做接单方落地页改造**：接单方（user 角色）落地页仍为订单广场，不在本次范围。
2. **不做 OrderPublishPage 完全废弃**：OrderPublishPage 作为降级路径保留（AI 解析失败时切换为手工模式，走独立表单页）。
3. **不做对话式订单管理**：已提交订单的查看、里程碑确认、验收等流程仍在现有"我的订单"页面完成，不搬进对话。
4. **不做对话历史搜索**：v1 不提供历史对话的全文搜索功能，仅支持按列表浏览和点击恢复。
5. **不做对话跨订单上下文**：每个对话会话独立，AI 不会跨对话引用其他订单的信息。

## 4. 用户故事

### 4.1 核心发单流程

**US-1**：作为客户，我希望落地后直接看到一个对话输入框而不是一堆统计卡片，以便我第一时间开始描述需求。

**US-2**：作为客户，我希望在对话框中用自然语言描述需求后，AI 逐步追问补全信息，以便我不需要事先准备完整的需求文档。

**US-3**：作为客户，我希望在对话过程中看到右侧逐步成型的订单草稿预览，以便我实时确认 AI 是否正确理解了我的需求。

**US-4**：作为客户，我希望可以直接点击右侧草稿中的某个模块卡片让 AI 重新生成该模块内容，以便精准修正而不必重新对话。

**US-5**：作为客户，我希望在对话中直接说"把预算改成8-10万"就能更新草稿，以便用最自然的方式调整内容。

**US-6**：作为客户，我希望结算模式、报价、接单方式等结构化配置也在对话中由 AI 逐步询问完成，以便我不需要单独填一个表单。

**US-7**：作为客户，我希望草稿确认后直接在右侧点击提交审核，以便整个发单流程在一个页面内闭环完成。

### 4.2 草稿与订单管理

**US-8**：作为客户，我希望左侧上方显示我所有未提交的对话草稿，以便我随时恢复之前没填完的订单。

**US-9**：作为客户，我希望左侧下方显示我已提交的历史订单，以便快速查看订单状态和跳转到订单详情。

**US-10**：作为客户，我希望点击某个历史对话草稿后，中间对话区和右侧预览区恢复到当时的对话上下文和草稿状态，以便无缝继续编辑。

**US-11**：作为客户，我希望提交后的订单从上方草稿列表消失、出现在下方历史订单列表，以便清晰区分"还没交出去"和"已经交出去"。

**US-12**：作为客户，我希望被运营打回的订单自动回到上方草稿列表，以便我恢复对话上下文修改后重新提交。

### 4.3 新客户与边缘场景

**US-13**：作为首次访问的新客户，我希望落地后对话区有占位提示告诉我"描述您的需求，AI 帮您生成订单"，以便我知道该做什么。

**US-14**：作为客户，我希望在对话过程中可以随时上传 Word/PDF 需求文档作为补充信息，以便 AI 结合文档更准确地生成草稿。

**US-15**：作为客户，当 AI 连续3轮未能获取有效信息时，我希望 AI 主动建议我切换为手工填写或上传文档，以便不卡在对话里浪费时间。

**US-16**：作为客户，我希望在"对话发单"Tab 和"我的工作台"Tab 之间自由切换，以便我在发单和查看订单进度之间灵活切换。

## 5. 功能需求

### 5.1 三栏整体布局

#### 5.1.1 布局结构

落地页采用三栏布局，顶部 Tab 切换：

```
┌──────────────────────────────────────────────────────────────┐
│  GlobalNav（全局导航栏，不变）                                  │
├──────────────────────────────────────────────────────────────┤
│  [对话发单]  [我的工作台]                    ← Tab 切换        │
├──────────┬──────────────────────────┬────────────────────────┤
│  左侧栏   │       中间对话区          │      右侧预览区         │
│          │                          │                        │
│ ┌──────┐ │  ┌──────────────────┐   │  ┌──────────────────┐  │
│ │草稿   │ │  │  对话消息流       │   │  │  订单草稿预览     │  │
│ │会话   │ │  │  (AI ↔ 客户)     │   │  │  (骨架模块卡片)  │  │
│ │列表   │ │  │                  │   │  │                  │  │
│ └──────┘ │  │  AI: ...          │   │  │  ▸ 项目概述      │  │
│ ┌──────┐ │  │  客户: ...        │   │  │  ▸ 需求说明      │  │
│ │历史   │ │  │  AI: ...         │   │  │  ▸ 交付物要求    │  │
│ │订单   │ │  │                  │   │  │  ▸ 排期与预算    │  │
│ │列表   │ │  └──────────────────┘   │  │  ▸ 接单要求      │  │
│ │      │ │  ┌──────────────────┐   │  │                  │  │
│ │      │ │  │  输入框 + 发送    │   │  │  [确认并提交]    │  │
│ │      │ │  └──────────────────┘   │  └──────────────────┘  │
│ └──────┘ │                          │                        │
└──────────┴──────────────────────────┴────────────────────────┘
```

#### 5.1.2 布局规格

| 区域 | 宽度占比 | 最小宽度 | 最大宽度 | 说明 |
|------|---------|---------|---------|------|
| 左侧栏 | 18% | 220px | 300px | 上下分栏：草稿会话（上）+ 历史订单（下） |
| 中间对话区 | 44% | 400px | 600px | 对话消息流 + 底部输入框 |
| 右侧预览区 | 38% | 320px | 480px | 订单草稿骨架模块卡片，默认隐藏 |

- 右侧预览区**默认不出现**（宽度为0），AI 判断信息充分后展开。
- 右侧展开时，中间对话区宽度等比收缩，不出现横向滚动。
- 左侧栏支持折叠/展开（点击边缘把手），折叠后仅显示图标。
- 响应式：屏幕宽度 < 1200px 时左侧栏自动折叠；< 900px 时右侧预览区以抽屉形式从右侧滑出。

#### 5.1.3 Tab 切换

- **"对话发单"Tab**（默认）：三栏对话式工作台，如上所述。
- **"我的工作台"Tab**：保留现有 CustomerWorkbench 的卡片式布局（统计卡片 + 订单列表 + 近期动态），作为客户的"进度总览"入口。
- Tab 状态持久化到 localStorage，客户下次进入恢复上次选择的 Tab。

### 5.2 左侧栏：草稿会话 + 历史订单

#### 5.2.1 上区：草稿会话列表

**展示内容**：客户所有未提交审核的对话草稿（订单状态为"新建"或"已打回"）。

**每条草稿展示信息**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 草稿标题 | AI 从对话首条消息自动提炼，或客户手动命名 | "电商小程序开发" |
| 最后编辑时间 | 相对时间 | "10分钟前" |
| 草稿状态 | 进行中 / 已打回 | 标签显示 |
| 信息完整度 | 进度条，AI 计算的各模块完整度 | ▓▓▓▓░░░░ 50% |

**交互**：

- **点击草稿**：中间对话区恢复该草稿的对话上下文，右侧预览区恢复草稿内容（如有）。
- **新建草稿**：左侧栏顶部"+"按钮，创建新对话，中间对话区清空为初始状态。
- **删除草稿**：右键菜单或草稿条目上的删除按钮，二次确认后删除（软删除，保留30天可恢复）。
- **草稿排序**：按最后编辑时间倒序排列。
- **草稿标题编辑**：双击标题可重命名。

**空状态**：新客户首次进入时，草稿会话区为空，显示引导文案"开始第一次对话 →"指向中间输入框。

#### 5.2.2 下区：历史订单列表

**展示内容**：客户已提交审核的订单（状态为"待审核"及之后的所有状态）。

**每条订单展示信息**：

| 字段 | 说明 | 示例 |
|------|------|------|
| 订单标题 | 订单主标题 | "AI智能问答系统开发" |
| 订单状态 | 状态标签（待审核/推广中/交付中/已验收/已结算等） | "推广中" |
| 提交时间 | 相对时间 | "3天前" |
| 来源标记 | 弱链接标识（hover显示"来源：对话草稿"） | ⓘ 图标 |

**交互**：

- **点击订单**：在对话页内以**只读模式**展开该订单（详见 §5.6 历史订单只读展开）。
- **查看全部**：左栏下方历史订单区顶部提供"查看全部 →"入口，点击跳转到"我的订单"页面（MyOrders）。
- **订单排序**：按提交时间倒序。
- **状态筛选**：顶部支持快速筛选（待审核 / 进行中 / 已完成），默认全部。

**空状态**：新客户首次进入时，历史订单区为空，显示"暂无历史订单"。

#### 5.2.3 草稿与订单的生命周期关系

草稿会话和历史订单是**同一实体在不同生命周期阶段的展示**，两区**互斥**，不会重复出现：

```
┌──────────────┐
│  草稿会话区   │  ← 订单状态为"新建"或"已打回"
│  （上方）     │     可编辑、可恢复对话
└──────┬───────┘
       │ 提交审核
       ▼
┌──────────────┐
│  历史订单区   │  ← 订单状态为"待审核"及之后
│  （下方）     │     只读展开 + 跳转完整详情
└──────┬───────┘
       │ 运营打回
       ▼
┌──────────────┐
│  草稿会话区   │  ← 回到上方，恢复对话上下文修改
│  （上方）     │
└──────────────┘
```

- **提交审核**：草稿从上方移除，出现在下方历史订单列表。
- **运营打回**：订单从下方移除，回到上方草稿列表，恢复对话上下文。草稿状态标记为"已打回"，客户可见打回原因。
- **来源弱链接**：历史订单条目上保留一个弱链接标记（ⓘ 图标），hover 时显示"来源：对话草稿 · [草稿标题]"。点击该标记不跳转，仅显示来源信息。

### 5.3 中间对话区：AI 对话发单

#### 5.3.1 对话区结构

中间对话区从上到下分为三部分：

1. **对话头部栏**（固定）：当前草稿标题 + 信息收集进度指示器（小型进度条+百分比）。只读模式下显示"已提交订单 · 只读模式"标识 + 订单状态标签。
2. **对话消息流**（可滚动）：AI 和客户的消息交替排列，AI 消息左对齐，客户消息右对齐。
3. **输入框区**（固定底部）：多行文本输入框 + 附件上传按钮 + 发送按钮。只读模式下隐藏输入框区。

#### 5.3.2 初始状态（新草稿）

新草稿尚未发送第一条消息时，中间区域采用"分类引导 + 聚焦输入"形态：

1. **引导标题**：显示"想发布什么任务？"及说明"选择任务方向并描述需求，AI 会结合分类逐步完善订单内容。不确定分类也没关系，可以直接开始。"
2. **一级任务分类**：展示运营端当前启用的一级任务分类，名称、图标、排序以运营配置为准。原型示例包含软件开发、平面设计、视频创作、文本创作、产品评测、营销推广、硬件订单。
3. **二级任务分类**：客户选择一级分类后，渐进展示该一级分类下启用的二级分类。二级分类为可选项，不选择也可开始对话。
4. **分类可跳过**：客户无需选择任何分类即可输入需求并发送，系统由 AI 根据需求内容自动识别任务类型。
5. **分类切换规则**：切换或清除一级分类时，系统同步清空原已选二级分类，避免分类路径不一致。
6. **聚焦输入框**：页面主体展示大尺寸多行输入框，支持文本输入、附件入口、点击发送和 Ctrl+Enter 发送。输入框底部展示当前分类路径；未选择分类时提示"AI 将根据描述自动判断分类"。
7. **对话消息流**：初始为空，不显示 AI 消息。AI 不主动开场。
8. **右侧预览区**：初始不出现；发送第一条消息后进入既有消息流，达到预览触发条件时再展开。

#### 5.3.3 对话交互流程

**客户发送第一条消息**：

1. 客户可预选一级分类、可选二级分类，也可完全跳过分类。
2. 客户在聚焦输入框输入需求描述，点击发送或按 Ctrl+Enter。
3. 系统创建草稿会话，将预选分类作为会话元数据写入，不将分类伪装成客户消息；分类作为 AI 首轮分析和运营骨架匹配的先验线索。
4. 页面收起初始分类选择区域，切换到现有对话消息流；对话头部显示分类路径（如"软件开发 / 小程序开发"）和信息收集进度。
5. AI 分析客户实际描述：
   - **未预选分类**：AI 根据需求内容自动识别任务类型。
   - **预选分类与描述一致**：AI 按该方向开始收集骨架信息，并在首轮回复中确认当前分类方向。
   - **预选分类与描述不一致或涉及多类型**：AI 可建议调整、增加或合并任务类型；最终类型以对话识别结果为准。
   - **信息不足**：AI 在对话流左侧回复追问问题（如"是移动端App还是微信小程序？""预算范围？""预期工期？"）。
   - **信息基本充分**：AI 回复"我已了解足够信息，为您生成订单预览"，右侧预览区展开。
   - **客户主动触发**：客户在任何时刻说"帮我生成订单"，AI 基于当前内容生成预览，信息不足的模块标记"信息待补充"。

**多轮对话**：

- AI 每轮回复后，对话头部栏的信息收集进度指示器更新（如 30% → 50% → 80%）。
- AI 追问覆盖的维度：以运营端为该订单匹配的任务类型所配置的骨架模块为准，逐模块评估信息覆盖度。同时覆盖跨类型的通用结构化必填项（结算模式、订单性质与接单方式等）。
- 结构化必填项（结算模式、报价、交付模式、接单要求、时限等）在对话中由 AI 逐步询问，客户回答后自动填入右侧草稿。

**信息充分度判断**：

AI 的信息充分度评估**不以固定的模块列表为准**，而是动态依赖运营端配置的骨架模板：

1. **模块来源**：客户预选的一级、二级分类用于首次加载对应任务类型的运营骨架并作为 AI 判断线索；AI 在对话过程中继续对实际订单内容做语义识别，确认或修正任务类型标签（参见客户发单 PRD §2.1.1 第3步）。最终任务类型确定后，系统读取运营端为该类型配置的**骨架模板模块列表**（参见《订单详情页弱结构化展示设计说明》§2.2 按任务类型可配置的骨架）。多类型标签的订单取所有匹配类型骨架模块的**并集**。
2. **覆盖度评估**：AI 对骨架模板中的每个模块逐一评估信息覆盖度，判断依据为该模块是否有足够信息生成有效内容。覆盖度评估逻辑由 AI 按模块语义动态判断，**不硬编码固定字段**。
3. **进度计算**：信息收集进度 = 已充分覆盖的模块数 ÷ 骨架模板模块总数 × 100%。对话头部栏的进度指示器实时更新。
4. **触发阈值**：当覆盖率 ≥ 80% 时，AI 主动提示"我已了解足够信息，为您生成订单预览"。
5. **提前触发**：客户可随时说"帮我生成订单"提前触发预览，未覆盖模块标记"信息待补充"并在右侧草稿高亮。
6. **类型变更**：若对话过程中 AI 重新分类了任务类型（如从"软件开发"调整为"产品评测"），骨架模块列表相应切换，已覆盖模块的信息保留映射到新骨架的对应模块，新增模块重新评估覆盖度。

#### 5.3.4 输入框能力

| 能力 | 说明 |
|------|------|
| 多行文本 | 支持换行，最大 2000 字 |
| 发送方式 | 点击发送按钮 或 Ctrl+Enter |
| 文档及多模态附件 | 点击附件按钮上传需求文档或图片等底层模型支持的材料，AI 解析后融入对话上下文；具体格式由接入能力约束 |
| 分类路径 | 新草稿输入框底部展示预选一级/二级分类；未选择时提示由 AI 自动判断 |
| 对话中修改 | 客户可输入"把预算改成8-10万""改为私密订单"等修改指令，AI 更新右侧草稿 |

#### 5.3.5 AI 异常处理

| 异常场景 | 处理方式 |
|---------|---------|
| AI 连续3轮未获取有效信息 | AI 主动建议"我可能需要更多信息才能帮您生成订单，要不要改用手工填写或上传文档？"并提供切换按钮 |
| AI 解析超时（>30秒） | 提示"AI 解析耗时较长，是否继续等待或切换为手工创建？" |
| AI 解析失败 | 提示"AI解析未能完成，已切换为手工创建模式"，保留已有对话内容，跳转 OrderPublishPage |
| 客户上传文档解析失败 | 提示"文档解析失败，请检查文件格式（支持 .docx/.pdf）或直接在对话框描述需求" |

### 5.4 右侧预览区：订单草稿

#### 5.4.1 出现时机

- **默认状态**：右侧预览区不出现（宽度为0）。
- **首次出现**：AI 判断信息充分并提示"已生成订单预览"时，右侧预览区从右侧滑入展开（300ms 动画）。
- **提前触发**：客户说"帮我生成订单"时，即使信息不充分也展开，未覆盖模块显示"信息待补充"占位。

#### 5.4.2 预览区结构

右侧预览区从上到下：

1. **预览头部栏**：标题"订单草稿预览" + 信息完整度百分比 + AI 自动分类的类型标签。
2. **骨架模块卡片列表**（可滚动）：按骨架顺序排列的模块卡片。
3. **底部操作区**（固定）："确认并提交审核"按钮（所有必填项满足后可点击）。

#### 5.4.3 骨架模块卡片

每个骨架模块以独立卡片展示。**卡片模块列表以运营端为该订单任务类型配置的骨架模板为准**（参见《订单详情页弱结构化展示设计说明》§2.2），不同任务类型展示的模块组合不同。以下为通用骨架模块示例（实际展示以运营配置为准）：

| 模块 | 内容形态 | 草稿预览展示 |
|------|---------|-------------|
| 项目概述 | Markdown | AI 生成的描述文本，Markdown 渲染 |
| 需求说明 | Markdown | AI 生成的需求详情，支持表格/列表 |
| 交付物要求 | Markdown | 交付物清单和格式要求 |
| 排期与预算 | 混合 | 结构化标签（预算区间、工期）+ Markdown 说明 |
| 接单要求 | 混合 | 结构化标签（资质要求）+ Markdown 说明 |
| 附加信息 | Markdown | 参考资料、备注 |
| 订单标签 | 结构化 | AI 生成的标签云 |
| 结算模式 | 结构化 | 平台抽佣 / 平台承包转包（标签） |
| 订单属性 | 结构化 | 私密 / 公开（标签） |
| 接单方式 | 结构化 | 指定接单方 / 公开抢单 / 推荐推送（标签+配置） |

**卡片交互**：

| 操作 | 行为 |
|------|------|
| 查看卡片 | 默认全部展开，支持折叠/展开 |
| 重新生成 | 卡片右上角"重新生成"按钮 → AI 以当前对话上下文重新生成该模块内容 |
| 内联编辑 | 点击卡片内容区域 → 进入内联编辑模式（Markdown 编辑器），修改后保存 |
| 对话调整 | 在中间对话区输入"把[模块名]改成…" → AI 更新对应卡片 |
| 信息待补充 | 未覆盖模块显示灰色占位"信息待补充"，高亮提示客户 |

#### 5.4.4 结构化配置项

以下结构化必填项在对话中由 AI 逐步询问，客户回答后填入右侧草稿对应区域：

| 字段 | AI 询问方式 | 示例 |
|------|-----------|------|
| 结算模式 | "您希望采用平台抽佣（您与接单方直接结算）还是平台承包转包（平台承接后转包）？" | 客户回答"平台抽佣" → 填入草稿 |
| 订单报价 | "该订单的预算范围是多少？" | 客户回答"5-8万" → 填入草稿 |
| 订单交付模式 | "交付方式是远程交付还是需要驻场？" | 客户回答"远程" → 填入草稿 |
| 接单要求 | "对接单方有什么资质或经验要求？" | 客户回答"3年以上React经验" → 填入草稿 |
| 订单性质 | "这个订单是公开抢单还是指定给某人？" | 客户回答"公开" → 填入草稿 |
| 接单方式 | 公开订单追问"要公开抢单还是推荐推送？" | 客户回答"推荐推送" → 展开推送配置 |
| 响应/招标时限 | "报名截止时间设多久？" | 客户回答"7天" → 填入草稿 |
| 超时处理 | "超时后自动下架还是转为公开抢单？" | 客户回答"转为公开抢单" → 填入草稿 |

**接单方式为"推荐推送"时的子流程**：

- AI 在对话中告知"根据您的订单标签，我为您匹配了以下优质接单方"，右侧草稿接单方式卡片展开匹配用户列表（Top 20，按标签重合度+信用分排序）。
- 客户在右侧卡片中勾选目标用户，设置推送时限。
- 确认后填入草稿。

#### 5.4.5 提交审核

- 所有必填项满足后，底部"确认并提交审核"按钮从禁用变为可点击。
- 客户点击 → 弹出确认弹窗，展示订单摘要（标题、类型、报价、接单方式）。
- 确认提交 → 订单状态从"新建"变为"待审核"，草稿从左栏上方移至下方历史订单列表。
- 提交成功 → 中间对话区显示"订单已提交审核"提示，右侧预览区收起。对话区恢复初始状态，准备下一次发单。

### 5.5 打回回退流程

当订单被运营打回时：

1. 订单状态从"待审核"变为"已打回"。
2. 订单从左栏下方"历史订单"列表移除，回到上方"草稿会话"列表。
3. 草稿状态标记为"已打回"，显示打回原因（运营填写，客户可见）。
4. 客户点击该草稿 → 中间对话区恢复提交时的对话上下文，右侧预览区恢复草稿内容。
5. 对话区顶部显示打回原因横幅："运营打回原因：[打回原因内容]，请修改后重新提交"。
6. 客户通过对话或内联编辑修改草稿内容。
7. 修改完成后点击"确认并提交审核"重新提交 → 订单状态从"已打回"变为"待审核"，草稿再次从上方移至下方历史订单列表。

**打回回退的边界规则**：

- 打回次数无限制，客户可多次修改重新提交。
- 每次打回都会回到草稿态，对话上下文保留全部历史（包括之前打回的记录）。
- 打回原因在草稿态和已提交态都可见。

### 5.6 历史订单只读展开

当客户点击左栏下方历史订单时，**不跳转页面**，而是在对话页内以只读模式展开该订单的历史对话和草稿快照。

#### 5.6.1 只读模式行为

| 区域 | 只读模式下的表现 |
|------|-----------------|
| 对话头部栏 | 显示"已提交订单 · 只读模式"标识 + 订单状态标签（待审核/推广中等） |
| 中间对话区 | 恢复该订单提交时的完整对话历史（只读，不可发送新消息） |
| 输入框区 | **隐藏**，不展示输入框和发送按钮 |
| 右侧预览区 | 展开，展示提交时的骨架模块快照（只读，不支持编辑/重新生成/提交） |
| 右侧底部操作区 | "确认并提交审核"按钮替换为"查看完整订单详情 →"按钮 |

#### 5.6.2 "查看完整订单详情"跳转

- 右侧预览区底部的"查看完整订单详情 →"按钮，点击后跳转到"我的订单"页面（MyOrders），并**自动打开该订单对应的详情弹窗**。
- 对话页内只读展示的是**提交时的骨架模块快照**（项目概述、需求说明、交付物等弱结构化内容）。
- 完整的订单详情（里程碑进度、协议签署、账单结算、报名情况等）在"我的订单"页面的订单详情弹窗中查看。

#### 5.6.3 只读模式与打回模式的切换

- **普通已提交订单**（待审核/推广中/交付中/已验收/已结算）→ 只读模式，不可修改。
- **已打回订单** → 自动切换为可编辑模式（恢复输入框 + 打回原因横幅 + 可修改重新提交），详见 §5.5 打回回退流程。

#### 5.6.4 退出只读模式

- 客户点击左栏上方任一草稿会话 → 退出只读模式，回到草稿编辑态。
- 客户点击左栏顶部"+"新建对话 → 退出只读模式，进入新草稿初始状态。

| 客户发布订单与审核 §2.6 运营审核操作 | **打回流程衔接** | 运营打回后触发本需求的"打回回退流程" |

#### 5.8.3 技术考量

- **对话状态管理**：对话上下文需要在客户端维护（或后端会话管理），切换草稿时需快速恢复。建议使用 sessionStorage 或后端 API 按需加载。
- **AI 接口**：需要后端提供对话式 AI 接口（多轮对话 + 信息充分度评估 + 草稿生成），现有 AI 解析接口需扩展为流式对话模式。
- **草稿持久化**：草稿需自动保存（每次对话轮次后保存），防止刷新丢失。建议 debounce 2秒自动保存到后端草稿表。
- **骨架模块渲染**：右侧预览区的模块卡片复用订单详情页的 Markdown 渲染组件，确保一致性。
- **分类配置读取**：一级、二级分类名称、层级、图标、排序与启停状态通过运营端任务类型配置接口读取，客户端不得将分类清单写死为业务规则；原型阶段可使用 mock 数据验证交互。
- **分类与AI接口**：首轮对话请求增加预选一级、二级分类参数；分类作为会话元数据传递，AI输出最终确认类型及类型来源，后端据此加载或切换骨架模板。

## 7. 非功能性需求

- **性能**：对话消息发送后 AI 首字响应时间 ≤ 2秒（正常网络条件）；右侧预览区展开动画 300ms 内完成；左栏草稿列表加载 ≤ 1秒。
- **安全**：仅已认证客户可进入对话式工作台；对话内容仅客户本人和运营可见；草稿自动保存频率 debounce 2秒。
- **兼容性**：支持 Chrome 90+、Firefox 88+、Edge 90+；响应式适配 ≥ 900px 宽度屏幕。
- **可用性**：初始分类可跳过，输入框始终可用；选择一级分类后渐进展示二级分类，不增加强制向导步骤；AI 异常时有明确的降级路径提示；草稿自动保存防止数据丢失。
- **可访问性**：一级、二级分类使用可键盘操作的单选语义并提供 aria-selected；动态出现的二级分类使用 aria-live="polite"；对话消息流支持键盘导航，输入框和发送按钮提供 aria-label。
