# 客户发布订单与审核产品需求说明书

## 需求概览

### 核心摘要
本需求将客户「AI驱动发单—审核—发布」的完整链路讲清楚、落下去。客户有三种发单入口：**AI对话创建**（以对话形式输入需求，AI自主判断信息是否充分、何时生成订单预览；一次性输入足够信息或逐轮追问本质上是同一对话的不同节奏）、**上传需求文档**（Word/PDF 解析）、**手工创建**（降级路径）。AI对话和文档解析生成的弱结构化内容均映射到各骨架模块，客户通过**卡片式确认界面**逐模块确认和编辑，补充结构化必填项后提交审核。AI对话同时作为**全局注入点**——在文档解析后的卡片确认、手工填写的任一环节均可随时唤起。AI解析同步生成多维订单标签。

### 设计思路
设计上坚持「对话为主线、AI自主判断信息充分度、文档解析和手工为补充、卡片确认、与骨架一致」：

- **AI对话创建**是主要的发单入口。客户在一个统一的对话界面中输入需求——可以一次性描述清楚（相当于"第一轮就提供足够信息"），也可以只给个方向、让AI逐轮追问。AI 自主判断当前信息是否足以生成各骨架模块的内容：信息足够时主动提示"我已了解足够信息，为您生成订单预览"，客户可接受预览或继续对话补充；信息不足时AI继续提问。
- **上传文档创建**适用于已有书面需求文档的场景，AI解析文档后进入同样的卡片确认流程。
- **AI对话作为全局注入点**：在文档解析后的卡片确认、手工填写的任意环节均可唤起对话，AI以当前已有内容为上下文继续对话。
- **手工创建**作为降级路径保留。
- 结构化必填项在确认完AI生成内容后以独立步骤补充。订单属性与接单方式的组合严格按方案约束。

### 历史实现参考
在设计过程中，我们严格依据《CSDN 订单管理系统整体方案》第 6.3 节「客户发布订单与审核」及第 4.1 节「订单发布与审核」「订单状态流转」的表述进行拆分，发单流程、订单必填信息、结算模式、响应时限与招标时限、上架状态与超时行为、订单属性与接单方式、订单状态与流转及审核操作的描述与方案一致。同时参考了同项目《订单发布模板与任务类型管理产品需求说明书》中任务类型与模板选择、三种创建模式与模板结合、必含字段（订单交付模式、订单报价、接单要求）及校验规则的设计，本需求中订单内容字段与模板字段统一，不重复定义格式与校验，保持与模板需求的一致性。本项目 `docs/history/` 下未发现「客户发布订单与审核」专项历史需求文档，以整体方案及同项目模板需求为设计来源；文档结构与验收准则（Given-When-Then 与 Gherkin）格式与同项目《用户注册与身份认证》《订单发布模板与任务类型管理》需求说明书保持一致。

## 1.3 背景和价值

**背景与痛点**：
客户在订单广场发单需要经历「选类型→选模板→选创建模式→填内容→设属性与接单方式→提交审核」的完整流程。若必填信息不统一（结算模式、订单交付模式、报价、接单要求、响应/招标时限等）、订单属性与接单方式不清晰，会导致审核标准不一、订单发布后展示与接单逻辑混乱；同时，指定接单方与公开抢单的超时行为、时限可调整时机若未明确，会影响客户与接单方的体验及运营处理口径。

**业务价值**：
1. **多种AI入口覆盖不同场景**：自然语言输入适合需求基本清晰的场景；上传文档适合有现成需求文档的场景；多轮对话适合需求模糊、需要AI协助梳理的场景。三种入口共用同一套骨架和确认流程。
2. **多轮对话降低发单门槛**：发单方不需要事先准备完整的需求文档或清晰的描述——AI 通过逐轮提问引导客户将模糊想法转化为结构化的订单内容。这尤其适合非技术背景的客户或复杂项目。
3. **卡片式确认保证质量**：AI 生成内容以模块卡片形式呈现（与订单详情页骨架一一对应），客户逐模块确认、编辑或补充，既保持AI的效率优势，又保留人工把控空间。
4. **多轮对话全局可唤**：在创建流程的任何环节（输入需求、卡片确认、手工填写），客户均可随时唤起 AI 对话——AI 以当前已有内容为上下文，针对性地提供帮助。
5. **手工创建作为降级路径**：AI解析失败时自动切换为手工模式并保留已解析内容供参考，确保发单流程不中断。
6. **AI标签自动生成**：AI解析时同步生成多维订单标签（技能、行业、交付特征等），为订单广场筛选和供需匹配提供精准依据。

# 第2章：功能需求

## 2.1 发单流程与订单创建

### 场景描述

**场景 1：AI对话创建（统一入口，覆盖"一次性说清"和"逐步追问"两种节奏）**

客户进入**AI对话创建**界面。这是一个统一的对话输入区——客户可以一次性描述完整需求，也可以只给个方向让AI追问。两者的区别仅仅是对话的"轮次"不同，不构成独立入口。**客户不需要事先选择订单类型——AI在解析完订单内容后自动匹配类型标签。**

**节奏A：一次性说清**
客户在输入区键入："我需要一个电商小程序，React技术栈，包含商品展示、购物车、支付对接，预算5-8万，2个月交付。"AI判断信息已基本覆盖各骨架模块，回复"我已了解足够信息，为您生成订单预览"，然后生成各模块的弱结构化内容，进入卡片式确认界面。

**节奏B：逐步追问**
客户在输入区键入："我想做一个电商系统。"AI判断信息不足，开始逐轮追问——"是移动端App还是微信小程序？""需要哪些核心功能？""预算范围？""预期工期？"——每轮客户自然语言回答，AI持续评估信息充分度。当信息覆盖各骨架模块时，AI主动提示生成预览。客户也可在任何一轮说"帮我生成订单"提前触发预览。

**统一体验**：两种节奏共用同一个对话界面、同一套记忆上下文、同一个"信息充分度判断→生成预览"的决策逻辑。对客户而言，区别只在于"我一句话说完"还是"我让AI多问几句"——不需要在进入前就选择不同入口。

**场景 2：上传文档AI创建（适用于有现成需求文档的场景）**
客户上传 Word/PDF 需求文档，AI 解析文档内容后进入卡片式确认界面。在确认过程中，客户可随时通过内嵌对话唤起AI辅助（如"请把需求说明模块里的性能要求展开描述"）。

**场景 3：手工创建（降级模式）**
当 AI 解析失败或客户主动选择手工创建时使用，按骨架模块逐项手动填写内容。手工填写时也可随时通过内嵌对话唤起AI辅助。

### AI对话作为全局注入点

AI对话能力不仅限于创建入口，在文档解析后的卡片确认、手工填写等任意环节均可随时唤起（触发方式同 v2.1 定义）。唤起后AI以当前上下文为对话起点。

### 基础多模态对话与内容生成能力

在不改变“对话收敛需求→生成订单草稿→客户确认→提交审核”主流程的前提下，AI对话入口补充以下基础能力：

1. **多模态附件理解**：客户可在初始态或多轮对话中上传图片及常见文档；模型结合附件和当前对话提取项目目标、范围、交付要求及待确认项，并将可用内容映射到订单骨架。
2. **可感知处理状态**：文本对话展示理解需求、结合上下文、组织回复等阶段；附件对话展示识别材料、提取信息、整理草稿等阶段。仅展示业务处理状态，不展示模型内部思维链。
3. **Markdown与代码**：AI回答支持基础Markdown和围栏代码块，代码块可复制但不在线执行，不建设代码沙箱。
4. **文件生成下载**：AI可根据对话、附件解析结果和订单草稿生成MD、TXT、DOCX、XLSX等结构简单的文件供客户下载；生成文件不替代平台内订单数据和确认步骤。
5. **模型中立**：后台可接入豆包等通用多模态国产模型，后续可替换其他模型；业务字段、前端消息结构和验收口径不得绑定单一厂商。

本能力不包含在线执行代码、复杂Office编辑、工具编排、沙箱执行或复杂Agent能力。详细的界面、支持格式、处理状态、异常降级、数据项、非功能要求及验收准则见《对话式工作台落地页产品需求说明书》v1.2。

### 2.1.1 基本事件流程

#### 主业务流程：AI对话创建

**前置条件**：操作人为已认证客户；后台 AI 对话服务可用；运营已配置各任务类型的识别特征。

**流程描述**：
1. **进入AI对话**：客户进入统一的**AI对话创建**界面——一个对话输入区。客户可一次性输入完整需求，也可只给个方向让AI追问。**客户无需事先选择订单类型。**
2. **AI逐步获取信息**：AI分析当前对话内容，评估各骨架模块的信息充分度：
   - 若信息足以覆盖主要模块 → AI提示「我已了解足够信息，为您生成订单预览」
   - 若信息不足 → AI逐轮追问，覆盖项目概览、需求细节、交付期望、排期预算、接单要求等维度
   - 客户可在任意时刻说"帮我生成订单"触发预览
3. **AI生成预览 + 自动分类**：AI汇总对话内容，生成各骨架模块的弱结构化内容。同步将订单内容与运营配置的**类型识别特征**做语义匹配，自动为订单打上类型标签（如同时匹配"软件开发"和"产品评测"）。客户在卡片确认界面可查看和调整类型标签。多类型标签的订单详情页骨架取所有匹配类型的模块并集。
4. **AI生成预览**：AI汇总对话内容，生成各骨架模块的弱结构化内容并同步生成订单标签。进入**卡片式确认界面**——每个骨架模块以独立卡片展示，客户可：
   - 点击"确认"通过某卡片内容
   - 编辑某卡片的 AI 生成内容（支持 Markdown 编辑或通过对话补充）
   - 点击"重新生成"让 AI 重新生成某卡片内容
   - 通过内嵌对话精准调整某模块内容
5. **补充结构化必填项**：所有模块卡片确认完成后，填写**结算模式**（二选一）、**订单交付模式**、**订单报价**、**接单要求**及**响应时限/招标时限**。
6. **设置订单属性与接单方式**：
   - 设置**订单性质**（私密/公开）
   - 若为私密订单：接单方式仅可选**指定接单方**或**指定由平台运营方接单**
   - 若为公开订单：接单方式可选**推荐推送**、**指定接单方**或**公开抢单**

   **推荐推送子流程**（公开订单专属）：
   a. 客户选择"推荐推送"后，系统自动展示**匹配用户列表**——基于订单AI标签与用户标签做交集查询，按标签重合度+信用分降序排列，展示Top 20
   b. 列表每项展示：用户名、信用分、技能标签（高亮匹配的标签）、行业标签、历史交付率、当前负载（进行中订单数）
   c. 客户可勾选目标用户（支持全选/多选），设置**推送时限**（常用预设：1天/3天/7天或被推送方确认即止）
   d. 客户点击"推送"，系统向勾选的用户发送推送，订单进入被推送用户的「推荐给你」区域
   e. 推送后，若在时限内无用户确认接单，订单可按配置**自动转为公开抢单**或**自动下架**
   - 若为**指定接单方**或**公开抢单**：流程与v2.2一致
7. **提交审核**：客户点击提交，校验通过后订单状态由「新建」→「待审核」。

#### 主业务流程：上传文档AI创建

**前置条件**：同上；客户有现成的需求文档（Word/PDF）。

**流程描述**：
1. **上传文档**：客户上传 Word/PDF 需求文档，点击"AI解析"。
2. **AI解析生成 + 自动分类**：后台解析文档内容并映射到各骨架模块，同时自动匹配类型标签。
4. 之后步骤与AI对话流程的第4-7步相同（卡片确认→结构化补充→属性设置→提交）。

**后置条件**：订单已提交，状态为**待审核**；订单详情页弱结构化内容已就绪，订单标签已生成。

**系统响应与提示**：
- AI生成预览：展示"我已了解足够信息，为您生成订单预览"（对话模式）或"AI已完成文档解析，请逐模块确认"（文档模式）
- 必填项缺失：提示具体错误项，不提交
- 提交成功：提示"订单已提交审核"

#### 扩展事件流程

- **保存草稿**：客户在未提交前可保存草稿，订单保持「新建」状态；再次进入可继续编辑。具体是否支持草稿保存及草稿有效期，原始需求未明确，**需补充确认**。
- **编辑后重提**：订单被运营打回（**已打回**）后，客户修改内容可再次提交，订单状态由「已打回」→「待审核」。

#### 异常事件流程

- **必填项缺失**：提交时校验，提示「请完善必填项」并定位到具体字段。
- **AI 解析失败**：提示「AI解析未能完成，已切换为手工创建模式，已解析的内容已为您保留」，自动切换为手工模式，客户可基于保留的内容继续填写或重新输入。
- **AI解析超时**（超过30秒）：提示「AI解析耗时较长，是否继续等待或切换为手工创建？」
- **AI对话卡住**：若AI连续3轮未能获得有效信息，AI主动提示「我可能需要更多信息才能帮您生成订单，要不要改用手工填写或上传文档？」
- **AI对话提前生成预览**：客户在任何时刻说“帮我生成订单”，AI基于当前对话内容生成订单预览。信息不足的模块标记为“信息待补充”，在卡片确认时高亮提示。
- **附件解析失败**：失败附件单独标记，可重新上传或重试；其他已成功附件、当前对话和订单草稿不得丢失。
- **AI回答生成失败**：保留客户输入及附件解析结果，提供重新生成，不要求客户重复上传文件。
- **Office文件生成失败**：提供重试，并允许降级生成MD或TXT；不影响继续对话和提交订单草稿。
- **指定接单方未选择用户**：若接单方式为指定接单方，须选择平台内某用户；未选择时提示「请选择接单方」。

## 2.2 订单必填信息与结算模式

### 场景描述

客户发布订单时须填写**结算模式**、**订单交付模式**、**订单报价**、**接单要求**及**响应时限/招标时限**（指定接单方订单为响应时限，公开抢单订单为招标时限）。**结算模式**二选一：**平台抽佣**（客户与用户直接交易结算，由客户定义该订单的里程碑、结算等）或**平台承包转包**（客户将订单承包给平台、平台转包给用户，由平台运营方定义该订单的里程碑、结算等）。具体字段含义、格式与校验规则在需求文档中定义，可与发布模板字段统一配置。

### 2.2.1 基本事件流程

#### 主业务流程

**前置条件**：客户已选择任务类型与创建模式，并完成订单内容填写或草稿生成（手工或 AI 解析）。

**流程描述**：
1. **结算模式**：客户在订单表单中选择**平台抽佣**或**平台承包转包**；系统记录该选择，用于后续里程碑与结算由谁定义的业务规则。
2. **订单交付模式、订单报价、接单要求**：客户按模板或表单项填写；系统按发布模板配置执行必填与校验（格式、长度、数值范围等）。
3. **响应时限/招标时限**：若接单方式为**指定接单方**，客户配置**响应时限**（常用预设如 3 天、7 天、15 天或**无限制**）；若接单方式为**公开抢单**，客户配置**招标时限**（同样支持预设或无限制）。
4. 上述必填项均通过校验后，客户方可提交审核。

**后置条件**：订单记录中结算模式、订单交付模式、订单报价、接单要求、响应时限/招标时限已持久化；后续审核、发布、里程碑定义方逻辑依赖结算模式。

**系统响应与提示**：
- 未选择结算模式：提示「请选择结算模式（平台抽佣或平台承包转包）」。
- 订单报价等不符合校验：提示具体规则（如「订单报价须大于 0」）。

#### 异常事件流程

- **结算模式未选**：提交时校验，提示「请选择结算模式」。
- **订单报价/接单要求等违反模板校验**：与《订单发布模板与任务类型管理》中模板字段校验一致，提示具体错误项并阻止提交。

### 2.2.2 需求波及分析

- **影响模块**：订单创建表单、订单存储、后续「里程碑确认与交付验收」中里程碑定义方（客户 vs 运营方）逻辑。
- **数据影响**：订单表必填字段；与发布模板字段统一配置，不重复定义格式与校验。
- **业务规则影响**：结算模式二选一且必填；订单交付模式、订单报价、接单要求为必填并与模板一致。
- **历史文档查阅记录**：同 2.1.3；本功能对应整体方案 6.3.2「订单必填信息」「结算模式」。

## 2.3 响应时限与招标时限的配置及调整

### 场景描述

客户新建订单时需配置——**指定接单方**订单配置**响应时限**（被指定方需在时限内响应或拒绝），**公开抢单**订单配置**招标时限**（订单在订单广场接受报名的时长）；支持常用时限预设（如 3 天、7 天、15 天等）或**无限制**。该时限可在订单处于**待审核**、**未上架**（推广中且未上架）、**推广中（已上架）**时由客户灵活调整，具体可调整次数与校验在需求文档中定义。

### 2.3.1 基本事件流程

#### 主业务流程

**前置条件**：订单已创建且接单方式为指定接单方或公开抢单；订单状态为**待审核**、**推广中（未上架）**或**推广中（已上架）**。

**流程描述**：
1. **新建时配置**：客户在发单时为指定接单方订单选择响应时限（如 7 天或无限制），为公开抢单订单选择招标时限（如 15 天或无限制）。
2. **调整时机**：客户在订单详情或订单管理入口，当订单状态为**待审核**、**未上架**（推广中且未上架）、**推广中（已上架）**时，可进入「修改响应时限」或「修改招标时限」进行修改。
3. **保存**：客户修改后保存，系统校验时限取值合法（在预设枚举内或无限制）；校验通过后更新订单的响应时限/招标时限。

**后置条件**：订单的响应时限或招标时限已更新；若已上架公开抢单订单修改招标时限，新时限自保存起生效（具体是否按「剩余时长」或「新的绝对截止时间」由业务确认，**此处信息不明确，需补充确认**）。

**系统响应与提示**：
- 若业务限制可调整次数：达到次数上限时提示「响应时限/招标时限已达最大调整次数，无法再修改」。
- 保存成功：提示「时限已更新」。

#### 扩展事件流程

- **可调整次数**：具体可调整次数与校验在需求文档中定义；若原始需求未明确，实现前需与业务确认（如每单最多调整 3 次）。

#### 异常事件流程

- **状态不允许调整**：订单已进入协议签署中、交付中等状态后，响应时限/招标时限不再开放调整，入口不展示或操作时提示「当前状态不可修改时限」。

### 2.3.2 需求波及分析

- **影响模块**：订单创建页、订单详情/订单管理（客户侧）、指定接单方响应倒计时/公开抢单招标倒计时逻辑。
- **数据影响**：订单表响应时限、招标时限字段及可选「调整次数」计数（若业务要求）。
- **业务规则影响**：允许调整的状态为待审核、未上架、推广中（已上架）；可调整次数与校验需与业务确认。
- **历史文档查阅记录**：同 2.1.3；本功能对应整体方案 6.3.2「响应时限与招标时限」、4.1「响应时限/招标时限可在…由客户灵活调整」。

## 2.4 上架状态与超时行为

### 场景描述

**推广中**阶段区分**未上架**/**已上架**：如指定接单方尚未确认为未上架，公开抢单已展示于订单广场为已上架。指定接单方订单可配置：时限内未响应或拒绝 → **自动下架**或**自动转为公开抢单**。公开抢单订单可配置：招标时限超时 → **自动下架**或**超时订单自动隐藏**（不再在广场展示），具体采用下架或隐藏在需求文档中确定。

### 2.4.1 基本事件流程

#### 主业务流程

**前置条件**：订单已审核通过，状态为**推广中**；系统按接单方式与业务规则判定**未上架**或**已上架**。

**流程描述**：
1. **未上架/已上架判定**：
   - **指定接单方**订单：指定方尚未确认接单或拒绝前为**未上架**；指定方确认接单后为**已上架**（对该订单而言已确定接单方）。
   - **公开抢单**订单：审核通过并展示于订单广场后为**已上架**；未展示前为**未上架**（通常审核通过即展示，故多数情况下为已上架）。
2. **指定接单方超时**：若被指定方在**响应时限**内未响应或拒绝，系统按订单配置执行**自动下架**（订单状态可由推广中→已取消或标记为下架）或**自动转为公开抢单**（接单方式变为公开抢单，按招标时限在订单广场展示，供用户报名/竞标）。
3. **公开抢单超时**：招标时限到期后，系统按订单配置执行**自动下架**或**超时订单自动隐藏**（订单不再在订单广场展示）；具体采用「下架」还是「隐藏」以与业务确认为准，**此处信息不明确，需补充确认**。

**后置条件**：超时后订单已按配置执行下架或转为公开抢单/隐藏；订单广场仅展示未超时且未下架/未隐藏的公开抢单订单。

**系统响应与提示**：
- 指定接单方超时后若转为公开抢单：客户与运营可见「已转为公开抢单，招标时限为 X」；订单在订单广场展示。
- 公开抢单超时后：订单在订单广场不再展示；客户在订单看板中可见订单状态或「已下架/已隐藏」标记。

#### 异常事件流程

- **定时任务延迟**：超时检测若依赖定时任务，可能存在分钟级延迟；若业务要求精确到秒，需在实现时说明。

### 2.4.2 需求波及分析

- **影响模块**：订单状态机、订单广场展示逻辑、指定接单方响应倒计时、公开抢单招标倒计时与超时任务。
- **数据影响**：订单状态、上架状态、接单方式（转为公开抢单时更新）、订单广场可见性。
- **业务规则影响**：未上架/已上架判定规则；指定接单方超时→自动下架或转为公开抢单；公开抢单超时→自动下架或自动隐藏。
- **历史文档查阅记录**：同 2.1.3；本功能对应整体方案 6.3.2「上架状态与超时行为」。

## 2.5 订单状态与流转

### 场景描述

订单状态包括**新建**、**待审核**、**已打回**、**推广中**（含未上架/已上架）、**协议签署中**、**交付中**、**已验收**、**待结算**、**已结算**、**已取消**、**已终止**。提交后由运营方审核；审核通过 → **推广中**，不通过 → **已打回**（客户修改后可再提交 → 待审核）。**待审核**、**推广中**可由客户或运营方取消 → **已取消**；取消为逻辑删除，报名记录保留。选标/确认接单后 → **协议签署中**；协议签署并上传至平台（自动关联订单）→ **交付中**。**交付中**若某里程碑超期未达成则开放客户终止权 → **已终止**；终止后结算以双方协议为准，由运营人工审核并按条款执行，平台不自动结算。指定接单方在响应时限内未响应或拒绝时，按配置自动下架或转为公开抢单；公开抢单招标时限超时后按配置自动下架或自动隐藏。全部里程碑通过 → **已验收** → **待结算** → **已结算**。

### 2.5.1 基本事件流程

#### 主业务流程

**前置条件**：订单已创建并提交（状态为待审核）或已处于后续任意状态。

**流程描述**（与整体方案第 4 章一致）：
1. **新建** → 客户提交审核 → **待审核**。
2. **待审核** → 运营通过 → **推广中**；运营打回 → **已打回**（客户可修改后再提交 → 待审核）。客户或运营取消 → **已取消**。
3. **推广中**：区分未上架/已上架；客户或运营取消 → **已取消**。指定接单方超时未响应或拒绝 → 按配置**自动下架**或**转为公开抢单**；公开抢单招标超时 → 按配置**自动下架**或**自动隐藏**。选标/确认接单 → **协议签署中**。
4. **协议签署中** → 协议签署并上传至平台（自动关联订单）→ **交付中**。
5. **交付中** → 全部里程碑通过 → **已验收**；某里程碑超期未达成则开放客户终止权，客户行使终止 → **已终止**。
6. **已验收** → **待结算** → **已结算**。
7. **已取消**：逻辑删除，报名记录保留。

**后置条件**：订单状态与整体方案定义一致；本需求重点覆盖至「审核通过→推广中」及「待审核/推广中取消」，其余状态在《订单广场与报名竞标》《里程碑确认与交付验收》等需求中细化。

**系统响应与提示**：
- 客户提交后：订单状态变更为待审核，客户端提示「订单已提交审核」。
- 运营打回：客户可见打回原因，订单状态为已打回。
- 取消订单：确认弹窗「确定取消该订单？取消后报名记录保留」；确认后状态为已取消。

#### 扩展事件流程

- **已打回后重提**：客户修改后再次提交，订单由已打回 → 待审核。

#### 异常事件流程

- **重复提交**：订单已为待审核时，不允许重复提交，或提示「订单已在审核中」。

### 2.5.2 需求波及分析

- **影响模块**：订单状态机、客户订单看板、运营审核与订单列表、订单广场展示、后续协议签署与交付流程。
- **数据影响**：订单状态字段、上架状态字段、逻辑删除标记。
- **业务规则影响**：完整状态枚举与流转与整体方案第 4 章一致；本需求文档不改变方案已定的状态定义。
- **历史文档查阅记录**：同 2.1.3；本功能对应整体方案 4.1「订单状态流转」、6.3.2「订单状态与流转」。

## 2.6 运营审核操作

### 场景描述

运营方对客户提交的订单进行**通过**或**打回**操作。打回时需填写原因，客户可见。审核通过后，系统按订单属性与接单方式处理：公开抢单的订单上架订单广场；指定接单方的订单仅对指定用户开放；运营方接单的订单由运营方在后台承接。

### 2.6.1 基本事件流程

#### 主业务流程

**前置条件**：操作人为运营方，已登录运营后台并具备订单审核权限；存在状态为**待审核**的订单。

**流程描述**：
1. **进入待审核列表**：运营在后台通过「订单管理」「待审核」或业务看板进入待审核订单列表。
2. **查看订单详情**：运营点击某订单查看完整内容（任务类型、模板、创建模式、结算模式、订单交付模式、订单报价、接单要求、响应时限/招标时限、订单性质、接单方式及指定接单方等）。
3. **审核操作**：
   - **通过**：运营点击「通过」，系统校验订单数据完整（必填项均已填、接单方式与指定接单方/运营方接单一致）；校验通过后订单状态由**待审核** → **推广中**。系统按订单属性与接单方式处理：**公开抢单**的订单上架订单广场；**指定接单方**的订单仅对指定用户开放（如「指定给我的订单」入口）；**指定由平台运营方接单**的订单出现在运营端待承接列表，由运营方在后台承接。
   - **打回**：运营点击「打回」，需填写**打回原因**（必填）；保存后订单状态由**待审核** → **已打回**，客户在订单看板或订单详情中可见该订单状态为「已打回」及打回原因。
4. **客户可见性**：打回原因在客户侧订单详情或通知中展示，客户可据此修改后重新提交。

**后置条件**：
- 通过：订单为**推广中**，公开抢单订单已在订单广场展示，指定接单方订单对指定用户可见，运营方接单订单在运营端可承接。
- 打回：订单为**已打回**，客户可见原因并可修改后再提交。

**系统响应与提示**：
- 打回时未填写原因：提示「请填写打回原因」，不提交打回操作。
- 审核通过成功：提示「审核通过，订单已发布」或 Toast；待审核列表该订单移除。
- 打回成功：提示「已打回」；客户端展示打回原因。

#### 扩展事件流程

- **批量审核**：原始需求未提及批量通过/打回，若业务需要可在后续需求中补充。

#### 异常事件流程

- **订单已被取消**：若客户或运营在审核前已取消订单，订单状态为已取消，运营端待审核列表不应再展示该订单，或展示时提示「订单已取消」。
- **权限不足**：非运营或未具备审核权限的用户无法访问待审核列表或执行通过/打回。

```gherkin
    And 订单状态变为「待审核」
    And 运营端待审核列表可见该订单

  Scenario: 结算模式与必填项必填
    Given 客户在发单页已选择任务类型与创建模式并完成内容填写
    When 客户未选择结算模式即点击提交
    Then 系统提示「请选择结算模式」或类似必填提示
    And 系统不提交订单
    When 客户未填写订单报价或接单要求或未配置响应时限/招标时限即提交
    Then 系统提示对应必填项未填
    And 系统不提交
    When 客户补充全部必填项后再次提交
    Then 提交成功

  Scenario: 指定接单方订单须配置响应时限及超时处理
    Given 客户选择接单方式为「指定接单方」并选择某用户
    When 客户未配置响应时限即点击提交
    Then 系统提示请配置响应时限，不提交
    When 客户未选择「未响应/拒绝时」的处理方式（自动下架或转为公开抢单）即提交
    Then 系统提示请选择超时处理方式，不提交
    When 客户配置响应时限及超时处理方式后提交
    Then 提交成功

  Scenario: 公开抢单订单须配置招标时限及超时处理
    Given 客户选择接单方式为「公开抢单」
    When 客户未配置招标时限或未选择超时处理即提交
    Then 系统提示请配置招标时限及超时处理，不提交
    When 客户配置招标时限及超时处理（自动下架或自动隐藏）后提交
    Then 提交成功

  Scenario: 订单性质与接单方式组合正确
    Given 客户设置订单性质为「私密订单」
    When 客户查看接单方式选项
    Then 系统仅展示「指定接单方」「指定由平台运营方接单」
    Given 客户设置订单性质为「公开订单」
    When 客户查看接单方式选项
    Then 系统展示「指定接单方」「公开抢单」

  Scenario: 运营打回须填写原因且客户可见
    Given 存在待审核订单
    And 运营具备审核权限
    When 运营点击「打回」
    And 运营未填写打回原因即确认
    Then 系统提示「请填写打回原因」
    And 打回操作不提交
    When 运营填写打回原因后确认打回
    Then 订单状态变为「已打回」
    And 客户在订单详情或订单看板可见该订单状态为「已打回」及打回原因

  Scenario: 审核通过后按接单方式处理
    Given 存在待审核的公开抢单订单
    When 运营点击「通过」
    Then 订单状态变为「推广中」
    And 该订单在订单广场展示
    Given 存在待审核的指定接单方订单
    When 运营点击「通过」
    Then 订单状态变为「推广中」
    And 该订单仅对指定用户开放（如指定给我的订单入口可见）
    Given 存在待审核的运营方接单订单（私密且指定由平台运营方接单）
    When 运营点击「通过」
    Then 订单状态变为「推广中」
    And 该订单在运营端待承接列表中可被运营方承接

  Scenario: 待审核或推广中可取消且为逻辑删除
    Given 订单状态为「待审核」或「推广中」
    When 客户或运营方执行取消订单操作并确认
    Then 订单状态变为「已取消」
    And 报名记录保留在数据库中
    And 取消为逻辑删除

  Scenario: 响应时限/招标时限可在规定状态下调整
    Given 订单状态为「待审核」或「推广中（未上架）」或「推广中（已上架）」
    When 客户在订单详情或订单管理入口修改响应时限或招标时限并保存
    Then 系统更新订单的响应时限或招标时限
    And 保存成功提示
    When 业务规则限制调整次数且该订单已达调整次数上限
    And 客户再次尝试修改时限
    Then 系统提示不可再修改或已达最大调整次数

  Scenario: 指定接单方超时后按配置执行
    Given 订单为指定接单方且已配置「未响应/拒绝时」为「自动转为公开抢单」
    And 已配置招标时限
    When 被指定方在响应时限内未响应也未拒绝
    Then 系统将订单接单方式变为公开抢单
    And 订单按招标时限在订单广场展示
    Given 订单为指定接单方且已配置「未响应/拒绝时」为「自动下架」
    When 被指定方在响应时限内未响应也未拒绝
    Then 订单被自动下架，不再接受报名

  Scenario: 公开抢单超时后按配置执行
    Given 订单为公开抢单且已配置超时后「自动隐藏」（或自动下架）
    When 招标时限到期
    Then 订单在订单广场不再展示
    And 客户在订单看板仍可见该订单及状态（已下架/已隐藏等）
```

## 2.10 非功能性需求

- **性能**：订单提交（含必填校验）响应时间建议不超过 3 秒（正常网络条件下）；AI 解析创建订单的解析请求与《订单发布模板与任务类型管理》一致，超时时间建议不超过 60 秒。运营端待审核列表加载响应时间建议不超过 3 秒。
- **安全**：仅已认证客户可发单；仅运营方具备订单审核权限；打回原因仅客户与运营可见，不向其他角色展示。
- **兼容性**：客户发单与订单管理支持整体方案约定的网页端与移动端；运营审核以网页端为主；兼容方案约定的浏览器与操作系统。
- **可用性**：必填项与校验规则与发布模板统一，提示文案明确（如「请选择结算模式」「请配置响应时限」）；审核打回原因客户必可见，便于修改后重提。

## 附录：推荐推送技术实现说明

> 本文档面向后端开发同学，说明"推荐推送"功能不需要引入推荐算法或机器学习，仅需简单数据库查询即可实现。

### 核心逻辑

"推荐推送"本质是**标签交集查询 + 信用分排序**，不涉及任何机器学习模型。整个流程由四个简单的SQL查询（或等价的ORM操作）组成：

### 查询1：客户侧匹配用户列表

```sql
-- 入参：order_tags = ['React 18', 'TypeScript', '政务系统', '远程优先']
--       exclude_user_ids = [] (排除已推送过的用户)
--       limit = 20

SELECT
    u.user_id,
    u.user_name,
    u.credit_score,
    u.history_delivery_rate,
    u.active_order_count,
    u.skill_tags,
    u.industry_tags,
    -- 计算标签重合度（交集大小）
    ARRAY_LENGTH(ARRAY_INTERSECT(u.skill_tags, order_skill_tags)) +
    ARRAY_LENGTH(ARRAY_INTERSECT(u.industry_tags, order_industry_tags))
    AS tag_overlap_score
FROM users u
WHERE
    -- 用户技能标签与订单技能标签有交集
    ARRAY_INTERSECT(u.skill_tags, order_skill_tags) IS NOT NULL
    AND u.credit_score >= 50  -- 信用分不低于50
    AND u.status = 'active'
    AND u.user_id NOT IN (exclude_user_ids)
ORDER BY
    tag_overlap_score DESC,
    u.credit_score DESC
LIMIT 20;
```

### 查询2：订单广场「推荐给你」——被推送的订单

```sql
-- 入参：current_user_id

SELECT o.*, p.push_time, p.push_deadline
FROM orders o
JOIN order_pushes p ON o.order_id = p.order_id
WHERE
    p.pushed_user_id = current_user_id
    AND o.order_status = 'promoting'
    AND o.accept_mode = 'recommended_push'
    AND p.push_deadline > NOW()
    AND p.status = 'pending'  -- 未被该用户确认/拒绝
ORDER BY p.push_time DESC;
```

### 查询3：订单广场「推荐给你」——标签匹配但未被推送的订单

```sql
-- 入参：current_user_id, user_tags

SELECT o.*
FROM orders o
WHERE
    o.order_status = 'promoting'
    AND o.accept_mode = 'open_bidding'  -- 公开抢单
    AND o.order_id NOT IN (
        SELECT order_id FROM order_pushes WHERE pushed_user_id = current_user_id
    )
    -- 用户标签与订单标签有交集
    AND ARRAY_INTERSECT(o.order_tags, user_skill_tags) IS NOT NULL
ORDER BY
    ARRAY_LENGTH(ARRAY_INTERSECT(o.order_tags, user_tags)) DESC,
    o.create_time DESC
LIMIT 10;
```

### 查询4：推送操作（写操作）

```sql
-- 客户勾选用户并点击"推送"时执行

INSERT INTO order_pushes (order_id, pushed_user_id, push_time, push_deadline, status)
VALUES
    (:order_id, :user_id_1, NOW(), :deadline, 'pending'),
    (:order_id, :user_id_2, NOW(), :deadline, 'pending'),
    -- ... 批量插入
;

-- 更新订单的推送超时处理配置
UPDATE orders SET
    push_deadline = :deadline,
    push_timeout_action = :action
WHERE order_id = :order_id;
```

### 所需数据表

仅需在现有订单表基础上加一张关联表：

```sql
CREATE TABLE order_pushes (
    id           BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_id     BIGINT NOT NULL,        -- 订单ID
    pushed_user_id BIGINT NOT NULL,      -- 被推送的用户ID
    push_time    DATETIME NOT NULL,      -- 推送时间
    push_deadline DATETIME NOT NULL,     -- 推送时限
    status       ENUM('pending','accepted','rejected','expired') DEFAULT 'pending',
    INDEX idx_user_order (pushed_user_id, order_id),
    INDEX idx_order (order_id)
);
```

### 复杂度评估

| 维度 | 评估 |
|------|------|
| 新增代码量 | 后端约 200-300 行（4个查询 + 1个推送接口） |
| 新增表 | 1 张关联表（order_pushes） |
| 算法依赖 | 无。仅标签数组交集 + 排序 |
| 性能影响 | 标签交集查询在标签数量<50时性能忽略不计；建议对标签字段建GIN索引 |
| 后续扩展 | 若未来需要更复杂的推荐，可在此表基础上引入排序模型，当前SQL查询逻辑不变 |
