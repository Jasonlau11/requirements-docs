# 开发工具配置与订单资源用量产品需求说明书

## 需求概览

### 核心摘要
本需求将「开发工具与订单」从松散关联升级为「配置—发放—用量」的闭环：运营在后台按任务类型或按订单配置**可选工具列表**与**必选工具**，接单方（用户）在**【我的订单】-【交付中的订单】**的订单详情中从可选列表中**选定一个或多个交付工具**（如 Cursor + Trae 协同交付），资源在**协议签署完成、订单进入交付中**之后发放，订单**已结算或已取消/已终止**后资源**自动失效**；交付中支持**切换交付工具**，切换后旧资源优先失效（下游做不到则保持可用），用量统计**订单交付周期内完整用量**、**按工具分别统计**，运营可按订单汇总或按工具维度查看消耗统计。这样既保证了资源与订单绑定清晰、发放与回收有据，又为多工具协同与后续计费分析提供了可追溯的用量基础。

### 设计思路
设计上坚持「运营控可选/必选、用户选工具、资源跟订单走、用量按工具拆」：运营控制的是「该订单可从哪些工具中选」及是否必选某一/多个工具，用户仅在交付中的订单详情内从可选列表中选择，不超出运营配置范围；资源发放时机与订单状态强绑定（交付中发放、已结算/已取消/已终止失效），避免资源滥用与成本外溢；同一订单支持多选多工具且交付中可切换，用量统计不因切换清零，按工具分别累计、完整周期汇总，与整体方案中「按订单统计资源用量、供运营配置与决策使用」一致。工具与资源的采购与来源规则（部署云、开发工具、设计/文档/音视频工具的来源与团购等）作为业务规则说明保留，具体采购流程、渠道码、授信在需求或商务文档中细化；本需求不扩展方案未提及的计费、结算与资源抵扣逻辑。

### 历史实现参考
本需求严格依据《CSDN 订单管理系统整体方案》第 6.6 节「开发工具配置与订单资源发放、用量统计」及第 4.1 节「开发工具与资源绑定」拆分，运营端配置可选/必选工具、用户端在交付中订单详情配置交付工具、资源发放时机与失效、交付中切换工具及用量按工具分别统计的表述与方案一致。本项目 `docs/history/` 目录不存在，未在本项目目录中找到开发工具/资源相关历史需求文档，以整体方案为唯一设计来源；文档结构与验收准则格式参考了同项目《订单发布模板与任务类型管理产品需求说明书》《用户注册与身份认证产品需求说明书》，保持需求文档风格与 Given-When-Then/Gherkin 验收方式、需求波及分析中的历史文档记录格式一致。

## 1.3 背景和价值

**背景与痛点**：  
订单承接方（用户）在执行交付时往往需要依赖各类开发工具、文档工具、音视频工具等；若工具与订单脱节，则资源发放、用量归集与成本管控无法在订单维度落地。同时，运营需要能够按任务类型或按订单控制「该订单可从哪些工具中选」及是否必选某一/多个工具，并在后台按订单、按工具查看实际消耗用量，为后续资源计费与运营分析提供基础。

**业务价值**：  
1. **工具与订单绑定清晰**：运营端配置订单的可选工具列表或必选工具，用户仅在交付中的订单详情中从可选列表选定一个或多个交付工具，保证资源发放与用量统计均按订单维度归集。  
2. **资源发放与回收有据**：资源在协议签署完成、订单进入交付中之后发放；订单已结算或已取消/已终止后资源自动失效，避免资源滥用与成本外溢。  
3. **多工具协同与用量可追溯**：同一订单支持多选多个工具（如 Cursor + Trae 协同交付），各工具分别统计用量；交付中可切换交付工具，用量统计订单交付周期内完整用量、按工具分别统计，运营可按订单汇总或按工具维度查看消耗统计。  
4. **采购与来源规则明确**：工具与资源遵循既定采购与来源规则（部署云、开发工具、设计/文档/音视频工具的来源与团购等），为商务与运营提供规则依据；具体采购流程、渠道码、授信与团购细节在需求或商务文档中细化。

# 第2章：功能需求

## 2.1 运营端开发工具配置

### 场景描述

**场景 1：按任务类型或按订单配置可选/必选工具**  
运营人员在后台为订单配置「可选工具列表」或「必选某一/多个工具」。配置可按**任务类型**或**按订单**进行，并与订单发布模板或订单属性关联；在需求文档中需明确配置层级与优先级（如订单级配置覆盖任务类型级配置，或反之）。运营控制的是「该订单可从哪些工具中选」以及是否允许多选、是否必选某一/多个指定工具。

**场景 2：可选与必选的关系**  
例如某软件开发订单，运营配置可选工具列表为「Cursor、Trae、某文档工具」，并配置「Trae 为必选」。则用户在该订单的交付中配置交付工具时，必须从 Cursor、Trae、某文档工具中选定至少包含 Trae 的一个或多个工具；若仅选 Cursor 未选 Trae，系统应校验并提示需包含必选工具。

**场景 3：与订单发布模板或订单属性的关联**  
配置可与订单发布模板或订单属性关联，使某类订单（如某任务类型下的所有订单、或采用某模板的订单）默认具备相同的可选/必选工具设置，减少逐单配置；具体关联方式与优先级在需求文档中明确。

### 2.1.1 基本事件流程

#### 主业务流程

**前置条件**：操作人为运营方，已登录运营后台并具备开发工具配置权限；系统已维护交付工具主数据（工具标识、名称、分类等，见 2.3）。

**流程描述**：  
1. **进入开发工具配置**：运营在后台通过「后台设置」「开发工具配置」或业务看板进入开发工具与订单的配置入口。  
2. **选择配置层级**：运营选择按**任务类型**或**按订单**进行配置（若两种均支持，需明确优先级：如订单级配置优先于任务类型级，当订单未单独配置时取任务类型级配置）。  
3. **配置可选工具列表**：运营为选定任务类型或订单配置「可选工具列表」，从系统已维护的交付工具中勾选一个或多个工具，保存后该订单（或该任务类型下的订单）在用户端配置交付工具时仅能在此列表中选定。  
4. **配置必选工具（可选）**：运营可进一步配置「必选某一/多个工具」；用户配置交付工具时须包含全部必选工具，并可再选其他可选工具。  
5. **保存**：运营点击保存，系统校验必选工具集合为可选工具列表的子集（若有）；校验通过则保存成功，配置生效。

**后置条件**：该任务类型或订单的可选工具列表、必选工具（若有）已持久化；用户在该订单进入交付中后，仅能从未选列表中选定交付工具，且若存在必选工具则须包含。

**系统响应与提示**：  
- 若必选工具未包含在可选工具列表中，保存时提示「必选工具须包含在可选工具列表中，请调整配置」。  
- 保存成功：提示「配置已保存」或 Toast。

#### 扩展事件流程

- **多选控制**：运营可配置该订单是否允许多选多个工具（若方案约定均允许多选，则无需单独配置项；若需按订单或任务类型控制，在需求文档中明确）。  
- **配置继承**：按任务类型配置时，该类型下新建订单是否默认继承该配置；按订单配置时，是否支持从任务类型或模板带入默认值再覆盖。  

#### 异常事件流程

- **权限不足**：非运营或未具备配置权限的用户访问开发工具配置入口时，系统提示「无权限」或不予展示。  
- **必选工具不在可选列表中**：保存时校验并提示，阻止保存直至调整。

## 2.2 用户端配置交付工具

### 场景描述

**场景 1：交付中订单详情内配置工具**  
用户承接订单并完成协议签署后，订单进入交付中。用户在**【我的订单】-【交付中的订单】**下看到该订单，点击查看订单详情后，可**配置此订单采用的交付工具**，从运营为该订单配置的**可选工具列表**中**选择一个或多个工具**（如软件开发订单可选 Cursor + Trae 协同交付）。若运营配置了必选某一工具，则用户须包含该工具，并可再选其他工具。

**场景 2：必选工具校验**  
某订单运营配置了「Trae 为必选」，用户若仅选择 Cursor 未选 Trae，系统应提示「请至少选择必选工具：Trae」，并阻止保存直至用户勾选 Trae。

**场景 3：多工具与用量**  
同一订单支持多选多个工具，各工具**分别统计用量**；运营看板支持按订单汇总与按工具维度拆开查看。用户配置完成后，后续资源发放与用量统计均按本次选定的工具集合处理。

### 2.2.1 基本事件流程

#### 主业务流程

**前置条件**：当前用户为订单接单方；订单状态为**交付中**；协议已签署并上传至平台；运营已为该订单（或该订单所属任务类型）配置可选工具列表。

**流程描述**：  
1. **入口**：用户进入**【我的订单】**，切换至**【交付中的订单】**页签，在列表中找到目标订单，**点击查看订单详情**。  
2. **进入配置交付工具**：在订单详情页中，提供「配置交付工具」或「开发工具/资源」相关入口，用户点击后进入该订单的交付工具配置界面。  
3. **展示可选工具列表**：系统根据运营配置，展示该订单的**可选工具列表**（仅展示运营配置的可选工具）；若存在**必选工具**，在界面上明确标注（如「必选」标签）。  
4. **用户选择**：用户从可选工具列表中**选择一个或多个工具**（多选以运营配置为准）；若存在必选工具，用户须勾选全部必选工具，方可保存。  
5. **保存**：用户点击保存，系统校验：所选工具均在可选工具列表中；若存在必选工具，则所选工具集合包含全部必选工具。校验通过则保存成功，该订单的交付工具配置生效；随后资源发放与用量统计按此配置执行。

**后置条件**：该订单已绑定用户选定的交付工具集合；用户可在订单详情或「开发工具/资源」入口查看并获取当前订单绑定的 API Key、激活码等（见 2.5）；用量按工具分别统计（见 2.8）。

**系统响应与提示**：  
- 若用户未选择任何工具且存在必选工具，或未勾选全部必选工具，提示「请至少选择必选工具：[必选工具名称]」，并阻止保存。  
- 保存成功：提示「交付工具已保存」或 Toast；界面可展示当前已选工具列表。

#### 扩展事件流程

- **首次进入交付中未配置**：订单刚进入交付中时，用户可能尚未配置交付工具；资源发放以「已配置的交付工具」为准，未配置前不发放或按业务规则处理（如提示用户先配置交付工具）。  
- **已配置后再次编辑**：用户可在交付中再次进入配置页增删所选工具，即「切换或增删」交付工具（见 2.7），保存后按新配置生效，旧资源优先失效。

#### 异常事件流程

- **非接单方访问**：非该订单接单方的用户无法为他人订单配置交付工具；若通过非法请求访问，后端校验并拒绝。  
- **订单非交付中**：订单未在交付中状态时，不展示或不可操作「配置交付工具」；若订单已结算或已取消/已终止，资源已失效，不再提供配置入口。  
- **可选列表为空**：若运营未配置可选工具列表或列表为空，用户端展示「暂无可选交付工具，请联系运营配置」或类似提示，不展示勾选列表。

## 2.5 资源发放时机与用户获取

### 场景描述

**场景 1：发放时机**  
资源在**协议签署完成并上传至平台**（订单进入**交付中**）之后发放。即：选标/确认接单后订单进入协议签署中，双方签署并将协议上传至平台且自动关联订单后，订单流转为交付中；此时触发资源发放逻辑，用户方可获取该订单绑定的 API Key、激活码等。

**场景 2：用户查看与获取**  
用户在**交付中的订单详情**或「**开发工具/资源**」入口查看并获取当前订单绑定的 API Key、激活码等。**具体形态以现有或规划中的资源发放能力为准**（如跳转至资源中心、在订单详情内嵌展示、或通过邮件/站内信下发等），本需求不规定具体 UI 形态，仅约定入口与时机。

### 2.5.1 基本事件流程

#### 主业务流程

**前置条件**：订单已进入**交付中**；用户已为该订单配置交付工具（若业务要求先配置再发放，则配置为前置条件；若支持先发放后补配置，则按实际规则）；下游资源发放能力可用。

**流程描述**：  
1. **发放触发**：订单状态由协议签署中流转为**交付中**时，系统根据该订单已配置的交付工具（或按默认/规则）触发资源发放；若用户尚未配置交付工具，则按业务规则决定是否延迟至用户配置后再发放，**此处需与业务确认**。  
2. **用户入口**：用户在**【我的订单】-【交付中的订单】**中点击订单进入订单详情，或在订单详情中点击「开发工具/资源」入口；系统展示当前订单绑定的资源信息（如 API Key、激活码等），**具体形态以现有或规划中的资源发放能力为准**。  
3. **资源展示**：用户可查看并复制/使用当前订单绑定的资源；资源仅对订单接单方可见，且仅在订单处于交付中且未失效时有效。

**后置条件**：用户可在约定入口查看并获取当前订单绑定的资源；资源与订单、所选交付工具关联，供交付使用。

**系统响应与提示**：  
- 若订单未在交付中，不展示资源或提示「当前订单未在交付中，暂无资源」。  
- 若资源发放依赖下游系统，下游不可用时，提示「资源加载中，请稍后再试」或与下游约定的一致提示。

#### 扩展事件流程

- **资源发放与配置顺序**：若要求用户先配置交付工具再发放，则未配置前展示「请先配置交付工具」；若支持先发放后配置，则进入交付中即发放默认或全部可选工具资源（若有），用户配置后可能涉及切换与旧资源失效（见 2.7）。**此处信息不明确，需补充确认**。  

#### 异常事件流程

- **下游资源系统异常**：资源发放或查询失败时，提示「资源暂不可用，请稍后再试」或与下游约定的一致提示，不阻塞用户查看订单详情。  
- **非接单方访问**：仅接单方可查看该订单的资源；客户或他人不可见。

## 2.6 资源失效与回收

### 场景描述

订单进入**已结算**或**已取消**（含**已终止**）后，该订单绑定的资源**自动失效/收回**。用户在此后不再能使用该订单的 API Key、激活码等进行新调用；下游计量系统不再接受该订单资源的用量上报（或已上报的用量仍保留用于统计，但资源不可再使用）。系统需在订单状态变更时触发资源回收或失效通知，与下游资源/计量系统配合完成。

### 2.6.1 基本事件流程

#### 主业务流程

**前置条件**：订单此前处于交付中且已发放资源；订单状态变更为**已结算**或**已取消**或**已终止**。

**流程描述**：  
1. **状态监听**：当订单状态由交付中（或已验收/待结算）流转为**已结算**，或由任意状态流转为**已取消**或**已终止**时，系统触发资源失效/回收逻辑。  
2. **失效处理**：系统通知下游资源发放/计量系统将该订单绑定的资源标记为失效或收回；或在本系统内将订单-资源绑定标记为已失效，后续用户端不再展示或允许使用。  
3. **用户侧**：用户在该订单详情或「开发工具/资源」入口查看时，若订单已结算/已取消/已终止，不再展示有效资源或提示「该订单已结束，资源已失效」。

**后置条件**：该订单绑定的资源已失效/收回，不可再用于新调用；用量统计仍保留订单交付周期内的历史数据（见 2.8）。

**系统响应与提示**：  
- 用户查看已结算/已取消/已终止订单的资源入口时，提示「该订单已结束，资源已失效」或仅展示历史用量不展示凭证。

#### 异常事件流程

- **下游无法即时失效**：若下游资源系统无法做到随时控制资源失效，方案约定「保持可用」；即本系统发起失效请求后，若下游暂不支持即时失效，则资源在下游侧可能仍可用一段时间，本系统仅尽到发起义务，具体以与下游约定为准。

### 2.6.2 需求波及分析

- **影响模块**：订单状态机、资源发放/回收对接、用户端订单详情与资源展示。  
- **数据影响**：订单-资源绑定状态（有效/已失效）；与下游系统的失效接口。  
- **业务规则影响**：已结算、已取消、已终止三种状态下资源均自动失效。  
- **历史文档查阅记录**：见 2.8.3。

## 2.7 交付中切换交付工具

### 场景描述

同一订单在交付中支持**切换或增删**所选交付工具。用户可再次进入订单详情的「配置交付工具」或「开发工具/资源」入口，修改已选工具（增删或更换）。**切换后之前工具的资源优先失效**；若下游系统无法做到随时控制资源失效，则**保持可用**。**用量统计**：该订单统计**订单交付周期内的完整用量**（含切换前后各工具用量），**按工具分别统计**；运营在后台可按订单或按工具查看消耗统计。

### 2.7.1 基本事件流程

#### 主业务流程

**前置条件**：订单处于**交付中**；用户已配置过交付工具；用户为订单接单方。

**流程描述**：  
1. **再次进入配置**：用户在订单详情中进入「配置交付工具」或等价入口，系统展示当前已选工具列表（与可选工具列表），用户可增删或更换工具后保存。  
2. **保存后生效**：保存后，新选定的工具集合生效；**之前所选、本次被取消勾选的工具**对应的资源，系统发起**优先失效**请求至下游；若下游无法做到随时控制资源失效，则**保持可用**（方案已明确）。  
3. **新工具资源**：本次新增加的工具，按 2.5 的发放逻辑为该订单发放对应资源（若下游支持按工具按订单发放）。  
4. **用量统计**：该订单的用量统计**不因切换而清空**，统计**订单交付周期内的完整用量**（含切换前后各工具用量），**按工具分别统计**；即：工具 A 在未切换前的用量 + 工具 B 在切换后的用量等，分别汇总到工具 A、工具 B 的维度，运营可按订单汇总或按工具维度查看（见 2.8）。

**后置条件**：订单当前绑定的交付工具集合已更新；旧工具资源已请求失效（或保持可用）；用量数据按工具分别保留完整周期内的统计。

**系统响应与提示**：  
- 保存时若未包含必选工具，同 2.2 校验并提示。  
- 保存成功后可提示「交付工具已更新，部分工具资源可能已失效」或与产品约定的一致文案。

#### 扩展事件流程

- **多次切换**：用户可在交付中多次切换工具；每次保存以最新选定的集合为准，此前被取消的工具资源每次均发起优先失效（或保持可用），用量始终按工具分别累计。

#### 异常事件流程

- **下游无法即时失效**：按方案约定，若下游做不到随时控制资源失效则保持可用；不在本需求中强制下游行为。

### 2.7.2 需求波及分析

- **影响模块**：用户端配置交付工具、资源发放/失效对接、用量统计（按工具分别、完整周期）。  
- **数据影响**：订单-工具绑定关系更新；用量统计表按工具维度累计。  
- **业务规则影响**：切换后旧资源优先失效，做不到则保持可用；用量统计完整周期、按工具分别统计。  
- **历史文档查阅记录**：见 2.8.3。

## 2.8 用量统计与运营查看（火山云侧）

### 场景描述

本期订单系统**不建设**用量统计与集中查询能力。开发工具资源的实际消耗用量（如 API 调用量、token 等）以火山云平台侧统计结果为准，运营需要查看用量时，直接前往火山云对应控制台按资源进行查看，不在订单管理系统内提供按订单或按工具的聚合查询页面。

### 2.8.1 基本事件流程

#### 主业务流程

**前置条件**：订单已绑定交付工具并产生过资源使用（或至少已配置工具）；火山云侧已具备用量查询能力且运营具备对应查看权限。

**流程描述**：  
1. **数据来源**：用量数据由火山云平台侧生成并维护，订单系统不再对接下游计量上报/查询接口进行汇聚存储。  
2. **运营查看路径**：运营在有查看需求时，登录火山云控制台，在对应资源或工具服务页面查询用量数据。  
3. **系统边界**：订单系统仅负责开发工具配置与资源发放/失效流程，不提供「用量统计」「消耗统计」相关后台页面与筛选能力。

**后置条件**：运营可在火山云侧完成用量查看；订单系统内无新增用量统计查询入口与聚合数据。

**系统响应与提示**：  
- 订单系统不展示用量统计数据，不新增「暂无用量数据」类页面提示；如需查看用量，运营按既定流程前往火山云平台。

#### 扩展事件流程

- **导出**：订单系统不提供用量报表导出能力；如需导出，以火山云平台侧能力为准。  
- **统计口径**：用量指标（如 API 调用次数、token 数等）及口径以火山云平台侧定义为准，订单系统不重复定义。

#### 异常事件流程

- **平台数据延迟**：若火山云侧用量数据存在延迟，以火山云平台展示结果与更新时间说明为准。  
- **订单已终止/已取消后的查看**：订单系统不承载该类用量展示；是否可回看历史用量取决于火山云平台能力与权限。

## 2.9 验收准则

### 2.9.1 验收准则表格

| 编号 | 场景描述 | Given（前置条件） | When（触发条件） | Then（预期结果） | And（附加验证） |
|------|----------|-------------------|-------------------|------------------|-----------------|
| AC-1 | 运营可配置订单可选工具列表与必选工具 | 运营已登录后台并具备开发工具配置权限，系统已维护交付工具主数据 | 运营按任务类型或按订单配置可选工具列表并保存 | 系统保存成功，该订单（或该类型订单）在用户端配置交付工具时仅能在此列表中选定 | 若配置了必选工具，必选工具须在可选列表中，否则保存时提示并阻止 |
| AC-2 | 用户仅在交付中订单详情配置交付工具 | 当前用户为某订单接单方，订单状态为交付中，运营已配置该订单可选工具列表 | 用户进入【我的订单】-【交付中的订单】并点击该订单查看详情，进入配置交付工具入口 | 系统展示该订单的可选工具列表，用户可从中选择一个或多个工具保存 | 若存在必选工具，用户未勾选全部必选工具时提示「请至少选择必选工具：xxx」并阻止保存 |
| AC-3 | 资源在协议签署完成、订单进入交付中后发放 | 订单已由协议签署中流转为交付中，用户已配置交付工具（若业务要求先配置再发放） | 用户在订单详情或「开发工具/资源」入口查看 | 用户可查看并获取当前订单绑定的 API Key、激活码等（具体形态以资源发放能力为准） | 非交付中或非接单方不可查看有效资源 |
| AC-4 | 订单已结算/已取消/已终止后资源自动失效 | 订单此前处于交付中且已发放资源，订单状态变更为已结算或已取消或已终止 | 系统触发资源失效逻辑；用户再次打开该订单资源入口 | 该订单绑定的资源已失效，用户端提示「该订单已结束，资源已失效」或仅展示历史用量 | 下游收到失效请求或本系统标记已失效（若下游无法即时失效则按方案保持可用） |
| AC-5 | 交付中可切换交付工具，订单系统不承载用量集中统计 | 订单处于交付中，用户已配置过交付工具，且已产生部分用量 | 用户再次进入配置交付工具，增删或更换工具后保存 | 新选工具集合生效，之前被取消的工具资源优先失效（做不到则保持可用） | 用量查看路径不在订单系统内，运营如需查看应前往火山云平台 |
| AC-6 | 运营通过火山云平台查看相关资源用量 | 订单系统未建设用量统计/消耗统计页面，运营具备火山云平台查看权限 | 运营需要查看资源用量并登录火山云平台对应控制台 | 运营可在火山云侧查看相关资源用量数据 | 订单管理系统内不提供按订单或按工具维度的集中查询入口 |
| AC-7 | 交付工具分类与可选列表展示正确 | 系统已维护交付工具主数据（含工具名称、分类），运营已为某订单配置可选工具列表 | 用户在该订单交付中配置交付工具时打开可选列表 | 仅展示运营配置的可选工具，可选工具按分类展示（如开发工具、文档工具等）；必选工具有明确标注 | 用户不能选择不在可选列表中的工具 |
| AC-8 | 非接单方不能为订单配置交付工具或查看资源 | 当前用户非某订单接单方，该订单处于交付中 | 该用户尝试通过接口或入口为该项目配置交付工具或查看该订单资源 | 系统拒绝并提示无权限或不予展示 | 仅接单方可配置该订单交付工具并查看该订单有效资源 |

### 2.9.2 Gherkin 格式验收准则

```gherkin
Feature: 开发工具配置与订单资源发放、用量统计

  Scenario: 运营可配置订单可选工具列表与必选工具
    Given 运营已登录后台并具备开发工具配置权限

    Then 系统保存成功
    And 该订单（或该类型订单）在用户端配置交付工具时仅能在此列表中选定
    When 运营配置必选工具且必选工具不在可选列表中
    And 运营保存
    Then 系统提示必选工具须在可选列表中并阻止保存

  Scenario: 用户仅在交付中订单详情配置交付工具
    Given 当前用户为某订单接单方
    And 订单状态为交付中
    And 运营已配置该订单可选工具列表
    When 用户进入【我的订单】-【交付中的订单】并点击该订单查看详情

    And 用户可从中选择一个或多个工具并保存
    When 存在必选工具且用户未勾选全部必选工具
    And 用户点击保存
    Then 系统提示「请至少选择必选工具：xxx」
    And 系统阻止保存

  Scenario: 资源在协议签署完成、订单进入交付中后发放
    Given 订单已由协议签署中流转为交付中
    And 用户已配置交付工具（若业务要求先配置再发放）
    When 用户在订单详情或「开发工具/资源」入口查看
    Then 用户可查看并获取当前订单绑定的 API Key、激活码等（具体形态以资源发放能力为准）
    And 非交付中或非接单方不可查看有效资源

  Scenario: 订单已结算/已取消/已终止后资源自动失效
    Given 订单此前处于交付中且已发放资源
    When 订单状态变更为已结算或已取消或已终止
    Then 系统触发资源失效逻辑
    When 用户再次打开该订单资源入口
    Then 该订单绑定的资源已失效
    And 用户端提示「该订单已结束，资源已失效」或仅展示历史用量

  Scenario: 交付中可切换交付工具，订单系统不承载用量集中统计
    Given 订单处于交付中
    And 用户已配置过交付工具且已产生部分用量
    When 用户再次进入配置交付工具并增删或更换工具后保存
    Then 新选工具集合生效
    And 之前被取消的工具资源优先失效（做不到则保持可用）
    And 订单系统内不新增该订单的用量集中统计能力
    When 运营需要查看该订单相关资源用量
    Then 运营应前往火山云平台查看

    Given 订单系统未建设用量统计/消耗统计页面

    When 运营需要查看资源用量并登录火山云平台对应控制台
    Then 运营可在火山云侧查看相关资源用量数据
    And 订单管理系统内不提供按订单或按工具维度的集中查询入口

  Scenario: 交付工具分类与可选列表展示正确
    Given 系统已维护交付工具主数据（含工具名称、分类）
    And 运营已为某订单配置可选工具列表
    When 用户在该订单交付中配置交付工具并打开可选列表
    Then 系统仅展示运营配置的可选工具

    And 必选工具有明确标注
    And 用户不能选择不在可选列表中的工具

  Scenario: 非接单方不能为订单配置交付工具或查看资源

    When 该用户尝试为该项目配置交付工具或查看该订单资源

    And 仅接单方可配置该订单交付工具并查看该订单有效资源
```

## 2.12 非功能性需求

- **性能**：运营端配置保存、用户端交付工具列表加载与保存、用量统计页查询响应时间建议不超过 3 秒（正常网络条件下）；资源发放与失效依赖下游系统，超时与重试策略与下游约定。  
- **安全**：开发工具配置仅运营方可操作；仅订单接单方可配置该订单交付工具并查看该订单有效资源；API Key、激活码等资源凭证展示需脱敏或按现有资源发放能力的安全规范处理。  
- **兼容性**：运营端以网页端为主；用户端支持整体方案约定的网页端与移动端；兼容方案约定的浏览器与操作系统。  
- **可用性**：下游资源发放或计量不可用时，本系统提示明确、不崩溃；用量统计无数据时展示「暂无用量数据」或 0，不报错。
