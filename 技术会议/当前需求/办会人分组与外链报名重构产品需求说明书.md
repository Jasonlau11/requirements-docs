# 办会人分组与外链报名重构产品需求说明书

## 需求概览

引入"办会人分组"概念，将办会人分为**商单**、**KA**、**SMB** 三类。商单办会人由 CSDN 商务签约后开通，免费使用快速创建会议能力并支持外链报名；KA（关键客户）办会人首次创建会议免费，自第二次起须按套餐付费，且支持外链报名；SMB（中小客户）办会人首次创建会议免费，自第二次起须按套餐付费，但**仅支持平台内报名**。同时，外链报名从"参会人填写问卷后跳转"重构为"参会人登录态下一键跳转"模式，并在报名记录中沉淀参会人账号信息。

### 1.3 背景和价值

**背景问题**：

1. **办会人能力缺乏分层**：现有办会人体系下，所有办会人享有相同的创建能力与限制，无法对 CSDN 商务合作的优质办会人提供差异化支持，也难以对自然注册办会人建立付费转化路径。
2. **外链报名转化路径冗长**：现有外链报名要求参会人填写完整问卷才能跳转外链 URL，在外链报名场景下问卷数据无业务价值，反而成为转化阻力；同时参会人账号信息无法沉淀到会议系统，与"以平台数据为根本"的运营复盘诉求冲突。
3. **报名渠道选项与办会人身份解耦**：现有"外链报名"对所有办会人开放，无法精细化管控外链配置能力，也不利于后续面向 SMB 用户的简化产品策略。

**业务价值**：
- **商单办会人**：CSDN 商务签约办会人获得完整的免费创建能力，叠加 TaoToken 引流与外链报名能力，强化合作粘性。
- **KA / SMB 办会人**：以"免费 1 次"为转化钩子，通过"¥699 单次"和"¥1,999 3次"两种套餐建立 SaaS 化付费路径；同时在注册环节增加设备校验，降低多手机号重复注册白嫖免费额度的风险。
- **外链报名简化**：一键跳转 + 强制登录模式降低外链转化阻力，沉淀账号数据为后续数据看板与二次营销提供基础。

**用户价值**：
- 商单办会人无需付费即可使用全部能力。
- KA / SMB 办会人首次免费，降低准入门槛；后续按需购买，套餐结构清晰。
- 参会人在外链报名时无需填写冗余信息，一键跳转体验更顺畅。

## 第2章：功能需求

### 2.1 办会人分组（CREATOR-GRP-01）

#### 场景描述

办会人账号在系统中归属于一个分组，分组由"商单 / KA / SMB"三选一的单一枚举值表示，决定该办会人可用的能力、收费规则和报名渠道选项。三个分组扁平并列，无层级关系。

#### 用户故事

| 编号 | 用户故事 |
|------|----------|
| US-CG-01 | 作为运营人员，我希望将一个办会人账号设置为"商单"，以便按 CSDN 商务合作提供差异化能力 |
| US-CG-02 | 作为运营人员，我希望将一个办会人账号设置为 KA 或 SMB，以便精细化控制其外链报名权限和付费规则 |
| US-CG-03 | 作为新注册办会人，我希望系统默认将我归为 SMB，以便立即开始使用平台能力（首次免费） |
| US-CG-04 | 作为 SMB 办会人，我希望运营后台可以手动将我升级为 KA，以便在我需要外链报名时获得相应能力 |

#### 需求规格

**2.1.1 分组结构**

| 分组 | 适用场景 | 区分价值 |
|------|----------|----------|
| 商单（commercial） | CSDN 商务签约合作的办会人 | 免费使用、全部能力、外链可用 |
| KA（ka） | 关键客户（行业头部或潜在高价值） | 首次免费、后续按套餐付费、可使用外链报名 |
| SMB（smb） | 中小客户（自然注册普通办会人） | 首次免费、后续按套餐付费、**仅平台内报名**；后台可升级为 KA |

**2.1.2 字段定义（用户账号扩展）**

在 `User_Account` 实体上新增字段（沿用 `docs/用户体系产品需求说明书.md` 的数据项定义）：

| 字段 | 类型 | 取值 | 默认值 | 说明 |
|------|------|------|--------|------|
| `creator_group` | 枚举 | `commercial` / `ka` / `smb` | `smb` | 办会人所属分组；单一字段，扁平化结构 |

**2.1.3 分组判定规则**

| 触发场景 | 判定规则 |
|----------|----------|
| 新用户注册 | 系统自动分配 `creator_group = smb` |
| 商务签约 | 运营后台手动将办会人调整为 `creator_group = commercial` |
| SMB → KA 升级 | 运营后台手动将办会人调整为 `creator_group = ka` |
| KA → SMB 降级 | 运营后台手动将办会人调整为 `creator_group = smb`；不触发历史会议的回退处理 |
| 商单 → 非 KA / SMB | 运营后台手动调整；调整后该办会人后续创建会议须按 KA / SMB 规则付费；历史会议不受影响 |

**2.1.4 创建会议时的分组前置检查**

办会人点击"快速创建会议"或"发起会议"入口时，系统在创建流程启动前**先判定其分组**，再决定后续路径：

| 分组 | 创建路径 | 收费 | TaoToken | 外链报名 |
|------|----------|------|----------|----------|
| 商单 | 标准创建流程 | 免费 | 可见、可勾选 | Step 08 可见，可选 |
| KA | 标准创建流程 | 首次免费；后续按套餐 | 可见、可勾选 | Step 08 可见，可选 |
| SMB | 标准创建流程 | 首次免费；后续按套餐 | 可见、可勾选 | Step 08 隐藏，仅平台内报名 |

**前后端校验**：分组判定由前端控制 UI 展示，后端必须在创建会议接口中**再次校验**办会人分组与请求参数的一致性：
- 后端拒绝 `creator_group = smb` 的办会人提交"外链报名"配置。
- 后端拒绝未购买或额度不足的 KA / SMB 办会人完成发布。

#### 验收标准

| 编号 | 验收标准 |
|------|----------|
| AC-CG-01 | 新注册办会人默认分组为 SMB，无需任何运营配置 |
| AC-CG-02 | 运营后台可查看并修改办会人的 `creator_group` 字段（commercial / ka / smb 三选一） |
| AC-CG-03 | 商单办会人进入创建会议流程不触发任何付费弹窗，免费创建 |
| AC-CG-04 | SMB 办会人进入创建会议流程的 Step 08 时，外链报名选项不可见；如通过 URL 参数或 API 强制传入外链报名，后端拒绝并返回明确错误 |
| AC-CG-05 | KA 办会人进入创建会议流程的 Step 08 时，外链报名选项可见，与商单办会人表现一致 |
| AC-CG-06 | 运营调整分组后，办会人下一次进入创建会议时分组立即生效；不需要重新登录 |

### 2.2 KA / SMB 办会人付费创建会议（CREATOR-PAY-01）

#### 场景描述

KA / SMB 办会人首次创建会议可免费使用一次，自第二次起须按"单次套餐"或"三次套餐"购买。免费额度以**注册手机号**为基础标识，每个手机号享有 1 次免费发布额度；为避免用户通过多个手机号在同一设备重复注册并反复获取免费额度，注册环节增加设备校验：同一设备最多注册 2 个账号。设备校验仅用于限制新账号注册数量，不限制同一设备登录多个已有账号，也不限制同一账号在不同设备登录。

#### 用户故事

| 编号 | 用户故事 |
|------|----------|
| US-PAY-01 | 作为 KA / SMB 办会人，我希望首次创建会议时获得 1 次免费机会，以便无门槛体验平台能力 |
| US-PAY-02 | 作为 KA / SMB 办会人，我希望免费额度用完后能选择"单次套餐"（¥699/次）或"三次套餐"（¥1,999/3次），以便按需购买 |
| US-PAY-03 | 作为 KA / SMB 办会人，我希望在创建会议配置完成、点击"发布官网"前被提示是否购买，以避免误操作或意外扣费 |
| US-PAY-04 | 作为 KA / SMB 办会人，我希望在购买弹窗中清晰看到套餐内容和价格，以便快速决策 |
| US-PAY-05 | 作为运营人员，我希望 KA / SMB 办会人的购买和扣费行为可追溯，以便后续对账与运营分析 |
| US-PAY-06 | 作为运营人员，我希望系统能识别同一设备重复注册多个账号的行为，以减少多手机号重复注册获取免费额度的情况 |
| US-PAY-07 | 作为 KA / SMB 办会人，我希望支付宝支付成功后额度立即到账，并能查看订单和额度明细，以便确认购买结果和剩余次数 |
| US-PAY-08 | 作为运营人员，我希望在一个后台中查询套餐订单、额度账户和额度流水，并登记退款结果，以便完成最小运营闭环 |

#### 需求规格

**2.2.1 免费额度规则**

| 规则项 | 内容 |
|--------|------|
| 免费额度 | 每个注册手机号 1 次 |
| 判定依据 | 办会人账号 `User_Account.mobile`（即注册手机号） |
| 保留规则 | 仍保留"首次免费创建 1 次会议"规则；用户免费额度用完后，再创建会议需购买套餐 |
| 新用户初始化 | 注册成功后按手机号初始化 1 次免费额度，后续通过额度账户和额度流水管理，不再实时统计会议数量反推 |
| 历史迁移 | 上线快照日前，同一手机号下任一普通办会人账号存在至少 1 场审核通过或正式发布的会议，则免费额度初始化为已使用；仅草稿、待审核、驳回或创建失败记录不消耗免费额度 |
| 兜底 | 免费额度以标准化手机号为归属键；同一手机号不可通过换账号、换设备或换浏览器重新取得免费额度。付费额度绑定 `user_id`，不在同手机号账号间共享 |

**2.2.2 套餐结构**

| 套餐 | 价格 | 额度 | 有效期 | 适用场景 |
|------|------|------|--------|----------|
| 单次套餐 | ¥699 | 1 次 | 购买后立即生效，无时间限制 | 偶尔办会、试水期 |
| 三次套餐 | ¥1,999 | 3 次 | 购买后立即生效，无时间限制 | 短期内有多场办会计划 |

套餐为固定商品，本期不建设套餐配置后台；商品编码、名称、价格和额度由系统预置。订单创建时保存套餐名称、价格、额度和办会人分组快照，后续套餐规则调整不影响历史订单。

**2.2.3 设备注册校验规则**

为避免用户通过多个手机号重复注册账号并反复获取首次免费额度，系统在**提交注册**时进行设备校验。该校验只约束"新账号注册"，不约束"账号登录"。

| 规则项 | 内容 |
|--------|------|
| 校验时机 | 用户在注册表单完成手机号验证码、协议勾选等必要信息后，点击"注册"提交时触发 |
| 校验对象 | 新注册账号；不对登录行为做设备数量限制 |
| 识别方式 | 综合使用 IP 地址、浏览器 User-Agent、设备指纹、操作系统信息、浏览器信息、Cookie / LocalStorage 标识等生成设备识别结果；具体字段以技术方案可稳定采集范围为准 |
| 注册上限 | 同一设备最多注册 2 个账号 |
| 超限处理 | 若系统识别当前设备已注册账号数量 ≥ 2，则阻止继续注册，不创建账号，不建立登录态 |
| 提示文案 | 当前设备注册账号数量已达上限，请使用已有账号登录或联系管理员 |
| 登录不受限 | 同一设备允许登录多个已有账号，不因注册上限触发登录拦截 |
| 记录留存 | 系统需记录设备识别结果、注册账号、注册手机号、注册 IP、注册时间、User-Agent 等信息，便于排查误拦截和风控复盘 |
| 隐私合规 | 设备信息采集应在隐私政策中覆盖，前端不展示具体设备指纹细节；仅用于注册风控与反滥用 |

> 设备识别不是强实名校验，存在误判与绕过可能；本期目标是提高批量注册白嫖成本，而不是完全杜绝所有异常注册。

**2.2.4 创建会议入口弹窗**

| 场景 | 弹窗类型 | 文案 |
|------|----------|------|

**2.2.6 套餐订单与在线支付**

本期仅复用现有推广业务使用的支付宝支付通道、当前收款商户主体及已有支付接入基础；套餐业务订单、支付结果处理和额度入账由本需求实现，不假设推广收银台已经提供统一订单、退款或补偿接口。

| 规则项 | 内容 |
|--------|------|
| 支付方式 | 首期仅支持支付宝，不提供微信支付、对公转账或其他支付方式 |
| 订单创建 | 用户选择套餐并点击“立即购买”后，后端按预置商品信息创建待支付订单；价格和额度以后端为准，不接受前端自定义金额 |
| 订单状态 | `待支付` / `已支付` / `已关闭` / `已取消`；退款状态独立记录为`未退款` / `处理中` / `已退款` / `退款失败` / `不可退款` |
| 支付时限 | 待支付订单 30 分钟内有效；超时关闭。用户重新购买时创建新订单 |
| 支付成功判定 | 以支付宝支付回调或主动查询结果为准，不以前端成功页为准 |
| 回调校验 | 校验签名、商户号、业务订单号和实付金额；重复回调不得重复发放额度 |
| 主动查单 | 支付页和支付返回页支持查询订单最新状态；用于用户刷新及回调未及时到达时的补偿 |
| 支付成功处理 | 订单更新为已支付、套餐额度入账、额度流水生成须保持一致；任一步骤失败均记录异常并执行补偿 |
| 支付后续接 | 返回原会议编辑页并刷新额度，提示“购买成功，已到账 X 次发布额度”；不自动提交审核，由用户再次点击发布 |
| 支付失败/取消 | 订单保持待支付或关闭，不增加额度；用户可重新发起支付或重新下单 |

订单至少记录：订单号、`user_id`、手机号快照、分组快照、套餐编码及名称、购买额度、订单金额、实付金额、支付方式、支付状态、支付宝交易号、来源会议 ID、创建时间、支付时间和关闭时间。

**2.2.7 额度账户与归属**

| 规则项 | 内容 |
|--------|------|
| 免费额度归属 | 绑定标准化注册手机号，每个手机号 1 次 |
| 付费额度归属 | 绑定购买账号 `user_id`，不绑定手机号，不支持账号间转移或共享 |
| 修改手机号 | 付费额度跟随原 `user_id` 保留；免费额度仍按新旧手机号历史使用记录进行校验，不因换绑重新发放 |
| 分组转为商单 | 剩余付费额度保留，商单期间创建会议不消耗套餐额度 |
| 商单转回 KA / SMB | 剩余付费额度恢复可用 |

额度账户至少记录免费额度状态、付费可用额度、冻结额度、累计购买额度和累计核销额度。账户余额不得为负数。

**2.2.8 额度冻结、核销与释放**

| 会议动作 | 额度处理 |
|----------|----------|
| KA / SMB 提交会议审核成功 | 原子冻结 1 次额度；可用额度同步减少 |
| 提交接口失败 | 不冻结；如已冻结则自动释放 |
| 审核通过、正式发布 | 将该会议冻结额度核销，不再次减少可用额度 |
| 审核驳回 | 释放该会议冻结额度，可用额度恢复 |
| 审核通过前主动撤回 | 释放该会议冻结额度 |
| 正式发布后下架、删除或取消 | 不返还额度 |
| 已发布会议修改后重新提交 | 不重复消耗，继续关联原核销记录 |
| 复制会议后发布新会议 | 新会议独立消耗 1 次额度 |

同一会议的冻结、核销和释放以 `meeting_id + 额度动作` 保证幂等；同一账号并发提交多场会议时，不得超额冻结。

每次额度变化均生成不可编辑、不可删除的流水，至少记录流水号、`user_id`、变动类型、可用及冻结额度变化、变动前后余额、关联订单、关联会议、操作来源、操作人、原因和发生时间。错误通过新增反向补偿流水修正，不修改历史记录。

#### 需求规格

**2.5.1 配置端调整**

| 规则项 | 内容 |
|--------|------|
| 外链报名可见性 | 仅商单办会人、非商单 KA 可见；非商单 SMB 隐藏 |
| 问卷配置关联 | 选择"外链报名"时，Step 07 报名问卷配置**隐藏**（不展示） |
| 外链 URL | 必填项，校验 http:// 或 https:// 开头 |
| 售票设置 | 选择"外链报名"时，Step 09 售票设置隐藏（与现有逻辑一致） |
| 配置后字段 | `registration_channel = external_link`、`external_link_url`、`registration_questionnaire = null/empty` |

**2.5.2 参会端重构**

| 规则项 | 内容 |
|--------|------|
| 按钮文案 | 【外链报名】（替代原"前往官网完成报名"，强调一键动作） |
| 按钮状态 | 始终可点击（无问卷必填校验） |
| 未登录点击 | 弹出注册/登录弹窗，登录成功后**自动**再次触发跳转 |
| 已登录点击 | 系统读取登录账号信息，跳转到办会人配置的外链 URL，**新页签打开** |
| 页面行为 | 跳转后当前页面不关闭、不跳转 |
| 重复点击 | 不限次数；点击均按当前登录态沉淀数据；后端按"同一会议+同一账号+同一渠道"去重，首次沉淀为准 |

**2.5.3 报名记录数据沉淀**

| 字段 | 数据来源 | 是否必有 | 沉淀规则 |
|------|----------|----------|----------|
| `mobile` | `User_Account.mobile` | 必有 | 直接取办会人账号手机号 |
| `email` | `User_Contact.email` | 可空 | 若办会人已填写且已验证则取，否则为 null |
| `display_name` | `User_Identity.real_name` 优先，否则 `User_Profile.nickname` | 必有 | 优先实名，无实名取昵称 |
| `nickname` | `User_Profile.nickname` | 必有 | 直接取账号昵称（默认值或自定义值） |
| `registration_channel` | 枚举：`platform` / `external_link` | 必有 | 外链报名时为 `external_link` |
| `registered_at` | 系统时间 | 必有 | 点击按钮的服务器时间 |
| `meeting_id` | 当前会议 | 必有 | 关联会议 |
| `user_id` | `User_Account.user_id` | 必有 | 关联办会人账号 |
| `questionnaire_data` | 复合 | 必空 | 外链报名不收集问卷，写为空对象 |

> 报名记录的结构与现有平台报名记录**复用同一张表**（或同结构的逻辑实体），仅在 `registration_channel` 字段上做区分；其他字段定义与现有逻辑兼容。

**2.5.4 去重与首次为准**

| 规则项 | 内容 |
|--------|------|
| 去重依据 | `meeting_id + user_id + registration_channel` 组合唯一 |
| 首次提交 | 沉淀一条记录，字段值取点击按钮时的账号快照 |
| 同账号重复点击 | 不创建新记录，不覆盖已有记录；按钮仍可点击，**不弹出任何提示**（避免打扰跳转体验） |
| 沉淀时机 | 点击按钮即沉淀（不依赖外链是否成功打开） |

**2.5.5 办会人端审核页**

| 规则项 | 内容 |
|--------|------|
| 可见性 | 与现有"平台报名"记录在同一列表展示 |
| 标识 | 列表中新增"报名来源"列：值为 `平台报名` 或 `外链报名` |
| 操作 | 审核通过/拒绝、数据导出、查看详情等操作与平台报名记录一致 |
| 字段展示 | 展示手机号、邮箱（如有）、姓名/昵称 |

**2.5.6 运营端查看页**

| 规则项 | 内容 |
|--------|------|
| 可见性 | 在运营端的"会议报名用户"列表中可查看外链报名用户 |
| 标识 | 列表中展示"报名来源"列 |
| 操作 | 与现有"平台报名"记录一致；支持按"报名来源"筛选 |
| 导出 | 导出 Excel 时包含"报名来源"列 |

## 第3章：本期不包含（Non-Goals）

1. **不建设复杂财务系统**：本期仅支持支付宝，不支持微信支付、对公转账或其他支付方式；不建设套餐配置后台、标准电子发票、在线退款申请及审核、人工调额和额度赠送，仅实现套餐发布所需的最小订单、支付、退款结果登记和额度闭环。
2. **不修改 TaoToken 业务逻辑**：沿用 `TaoToken引流集成产品需求说明书.md`，所有办会人分组均按现有规则享受 TaoToken 赠送能力。
3. **不修改会议审核主流程**：仅在"审核通过"状态变更时新增短信提示；审核本身的状态机、规则、入口保持不变。
4. **不修改站内信审核通知**：现有站内信审核结果通知逻辑保持不变，与新短信并存。
5. **不修改运营端现有的"会议报名用户"页面结构**：仅新增"报名来源"列与按"报名来源"筛选能力，不重做列表布局。
6. **不追踪外链页面的后续行为**：参会人跳转外链 URL 后的注册/支付等行为不在会议系统追踪范围；外链点击数/转化数等数据由办会人自行在外链系统查看。
7. **不开放外链报名审核结果与外部页面联动**：外链报名记录的状态变更（通过/拒绝）不影响外链页面的实际报名状态；该限制沿用 `报名渠道配置产品需求说明书.md` 原 2.1.4 条款。
8. **不建设独立会员中心**：仅在“我的会议—我创建的会议”增加轻量“我的套餐”入口，承载额度概览、订单和额度明细，不扩展会员等级、自动续费、优惠券或套餐营销能力。
9. **不支持额度转赠或跨账号共享**：付费额度绑定购买账号 `user_id`，不支持转移、合并或拆分。

## 第7章：风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| SMB 办会人通过 API/URL 参数绕过前端限制 | 数据合规与计费口径错乱 | 后端二次校验，对 `creator_group = smb` 的请求拒绝外链配置 |
| 非商单办会人通过多手机号/多账号薅免费额度 | 付费转化漏斗失效 | 注册环节增加设备校验，同一设备最多注册 2 个账号；免费额度仍按注册手机号判断，且同一手机号不可通过换设备、换浏览器重置免费额度 |
| 外链报名强制登录影响转化 | 跳转路径增加一步，转化漏斗变长 | 通过弹窗文案与快速登录体验降低阻力；上线后监控外链点击率与登录率 |
| 报名记录数据结构不兼容外链场景 | 需新建表或大改表结构 | 复用现有报名记录表，仅新增 `registration_channel` 字段区分来源；数据层最小化改动 |
| 商单办会人解约后回退处理不当 | 存量会议可能因分组变化导致后续操作异常 | 明确分组变更仅影响后续创建，不回退历史会议；剩余额度保留，转回 KA / SMB 后恢复可用；运营后台提供分组变更审计日志 |
| 支付渠道已成功但业务订单未更新 | 用户已付款但额度未到账，引发投诉 | 支持支付回调和主动查单；以渠道最终状态补偿订单、额度和流水，重复处理保持幂等 |
| 并发提交导致额度超扣 | 同一账号可用额度出现负数或多场会议同时发布 | 提交审核时原子冻结额度，账户使用版本控制；同一会议按动作幂等 |
| 历史会议迁移误判免费额度 | 老用户被错误收费或重复获得免费额度 | 迁移前生成快照，按标准化手机号和历史已通过/已发布状态判断；误判通过反向调额流水修正 |
