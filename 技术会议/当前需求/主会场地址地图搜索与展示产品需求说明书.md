# 主会场地址地图搜索与展示产品需求说明书

## 需求概览

将创建会议流程中城市站的主会场地址输入方式，从纯文本输入框改为两步式地址填写：第一步通过高德地图 API 搜索并选择 POI 地址（严格限定在所选城市范围内），第二步填写门牌号、楼层、厅名、会议室名等详细位置信息。系统存储结构化地址数据（含经纬度坐标与详细位置补充信息）。参会人在会议详情页底部看到该城市站主会场地址的静态地图标注图，根据所选城市站动态切换展示；地图定位仍以办会人第一步选中的 POI 坐标为准。

---

## 第1章：概述

### 1.1 术语表

| 名称 | 详细描述 |
|------|----------|
| POI（Point of Interest） | 地图数据中的兴趣点，如"上海国际会议中心"、"浦东嘉里大酒店"等具体地点 |
| Place ID | 高德地图为每个 POI 分配的唯一标识，用于查询地点详情 |
| 结构化地址数据 | JSON 格式存储的地址信息，含完整文本地址、经纬度坐标、城市、区县、Place ID、详细位置补充信息 |
| 详细位置 | 办会人选中 POI 后补充填写的门牌号、楼层、厅名、会议室名等信息，如“3楼宴会厅A”“T2栋1601会议室” |
| 静态地图 | 一张标注了某个坐标点的地图图片（不可交互缩放/拖拽），通过高德静态图 API 生成 |

### 1.2 修订记录

| 版本号 | 内容 | 负责人 | 更新时间 |
|------|------|------|------|
| V1.1 | 增加详细位置字段，地址填写调整为“选择地址 + 填写门牌号/大厅名称”两步式；地图展示仍以所选 POI 坐标为准 | — | 2026-07-05 |
| V1.0 | 首版，主会场地址从文本框改为地图搜索+静态图展示 | — | 2026-06-22 |

### 1.3 背景和价值

- **背景问题**：当前创建会议时每个城市站的主会场地址是一个纯文本输入框。办会人手动输入地址后，参会人在详情页看到的是纯文字——既不能直观判断地点位置，也无法直接打开地图导航。对于巡回多城市场景，参会人需要自行复制地址去地图 App 中搜索，体验割裂。
- **业务价值**：提升会议详情页的信息辨识度和专业感，减少参会人"找不到会场"的咨询量。
- **用户价值**：办会人通过搜索选地址而非手打，减少填写错误和格式不一致；参会人一眼就能定位会场位置，降低到场阻力。

### 1.4 影响范围

| 端 | 影响 |
|----|------|
| 办会人创建端 | 城市站下的主会场地址输入框替换为两步式地址填写组件：先搜索选择地图地址，再填写门牌号/大厅名称等详细位置；后端存储格式从纯文本改为结构化 JSON |
| 参会人详情端 | 会议详情页底部新增静态地图展示区域；根据城市站切换联动更新 |
| 已有数据兼容 | 历史会议纯文本地址不做迁移，继续以文本形式展示；新创建/编辑的会议使用结构化存储 |

### 2.2 办会人端地址填写（MAP-01）

#### 场景描述

办会人在创建/编辑会议的"时间·地点·议程"步骤中，为每个城市站（限"线下"或"线上+线下"类型）配置主会场地址。地址填写分为两步：第一步选择地址，办会人在地址搜索框中输入关键词（如"国际会议中心"），系统调用高德地图 POI 搜索接口，返回结果按规则排列后以下拉列表展示；第二步填写详细位置，办会人补充门牌号、楼层、厅名、会议室名等信息（如"3楼宴会厅A"）。办会人完成两步填写后，系统存储该地址的结构化数据。

#### 用户故事

| 编号 | 用户故事 |
|------|----------|
| US-MAP-01 | 作为办会人，我希望通过输入关键词搜索城市范围内的地点，并从下拉列表中选中准确地址，以便准确且快速地填写主会场信息 |
| US-MAP-02 | 作为办会人，我希望搜索结果严格限定在所选城市内，不被其他城市的同名地点干扰 |
| US-MAP-03 | 作为办会人，我希望在选中酒店、会议中心等地图地址后继续填写门牌号、楼层、大厅名称或会议室名称，以便参会人到达建筑后能准确找到具体会场 |

#### 需求规格

**2.2.1 地图 API 选型**

选用 **高德地图 Web API**，涉及两个接口：

| 接口 | 用途 | 文档路径 |
|------|------|----------|
| POI 关键字搜索 | 根据办会人输入的关键词 + 城市限定，返回匹配的 POI 列表 | `/v3/place/text` |
| 静态地图 | 根据经纬度坐标生成标注了一点的静态地图图片 | `/v3/staticmap` |

> **选择理由**：无需 SDK 接入，纯 HTTP API 即可。申请一个 Web 服务 Key 后两端（创建端/参会端）均可直接调用。文档成熟，社区资源丰富。

**2.2.2 配置端交互**

地址填写组件分为两步：

| 步骤 | 字段 | 说明 |
|------|------|------|
| 第一步 | 选择地址 | 通过地图搜索选择酒店、会议中心、园区、写字楼等 POI 地址，决定地图定位坐标 |
| 第二步 | 填写门牌号 | 补充门牌号、楼层、大厅名称、会议室名称等具体位置信息，帮助参会人到达建筑后找到具体会场 |

```
办会人进入城市站 → 选择会议形式为"线下"或"线上+线下"
  ↓
主会场地址字段展示为两步式地址填写组件
  ↓
第一步：选择地址
  - 展示地图搜索输入框（带搜索图标和 placeholder）
  - 办会人输入关键词（≥2 个字符后触发搜索）
  - 300ms 防抖
  - 调用高德 POI 搜索 API，传入 city=所选城市、keywords=输入内容
  ↓
下拉列表展示搜索结果：
  - 每条结果展示：POI 名称（主标题）+ 行政区/详细地址（副标题）
  - 严格限定在该城市范围内，非该城市的结果不展示
  - 最多展示 10 条
  - 无结果时展示提示："未找到匹配的地址，请尝试其他关键词"
  ↓
办会人点击某条结果 → 下拉列表收起，输入框展示选中地址的文本
  ↓
第二步：填写门牌号
  - 展示详细位置输入框
  - placeholder 建议："请输入门牌号/楼层/大厅名称，如：3楼宴会厅A"
  - 办会人填写门牌号、楼层、大厅名称、会议室名称等信息
  ↓
系统存储结构化数据（见 2.2.4）
```

**2.2.3 搜索过滤规则**

| 规则项 | 内容 |
|--------|------|
| 城市限定 | **严格过滤。** API 调用时传入 `city` 参数，仅返回该城市范围内的 POI；非该城市的结果完全不展示 |
| 触发条件 | 办会人输入 ≥ 2 个字符后触发搜索；< 2 个字符不触发 |
| 防抖 | 300ms 防抖，避免每次按键都发起请求 |
| 请求超时 | 5 秒超时；超时后展示提示"搜索超时，请重试" |
| API 异常 | 接口返回异常时展示提示"地址搜索服务暂时不可用，请稍后重试" |
| 结果数量 | 最多展示 10 条；若实际返回 > 10 条，截取前 10 条 |
| 无结果 | 展示提示"未找到匹配的地址，请尝试其他关键词" |

**2.2.4 存储数据格式**

选中 POI 并填写详细位置后，系统以结构化 JSON 存储：

```json
{
  "text": "上海市浦东新区世纪大道100号 上海环球金融中心",
  "detail": "3楼宴会厅A",
  "display_text": "上海市浦东新区世纪大道100号 上海环球金融中心 3楼宴会厅A",
  "lat": 31.2363,
  "lng": 121.5051,
  "place_id": "B0FFxxxxxxxxxxx",
  "city": "上海市",
  "district": "浦东新区"
}
```

| 字段 | 类型 | 说明 | 来源 |
|------|------|------|------|
| `text` | String | 地图地址文本，即第一步选中的 POI 地址 | 高德返回的 `name` + `address` 拼接 |
| `detail` | String | 详细位置补充信息，如门牌号、楼层、大厅名称、会议室名称 | 办会人手动填写 |
| `display_text` | String | 面向参会人展示的完整地址文本，建议由 `text` + `detail` 拼接生成 | 系统生成 |
| `lat` | Float | 纬度 | 高德返回的 `location` |
| `lng` | Float | 经度 | 高德返回的 `location` |
| `place_id` | String | POI 唯一标识 | 高德返回的 `id` |
| `city` | String | 所属城市 | 高德返回的 `cityname` |
| `district` | String | 所属区县 | 高德返回的 `adname` |

> 地图定位和静态地图标注点仅使用 `lat` + `lng`，即第一步选中的 POI 坐标；`detail` 仅用于文本展示，不参与地图坐标计算。

**2.2.5 详细位置填写规则**

| 规则项 | 内容 |
|--------|------|
| 字段名称 | 建议命名为“门牌号/大厅名称”或“详细位置”，具体以设计稿为准 |
| 填写时机 | 办会人完成第一步 POI 地址选择后展示并填写 |
| 必填规则 | 线下、线上+线下城市站中，建议作为必填字段；若需兼容部分无门牌号场景，可允许填写“无”或“详见现场指引” |
| 输入内容 | 支持中文、英文、数字及常见符号，用于填写门牌号、楼层、厅名、会议室名等 |
| 字数限制 | 建议 2-50 字，超出后提示“详细位置最多输入50个字” |
| 示例文案 | 3楼宴会厅A、T2栋1601会议室、负一层多功能厅、A座报告厅 |
| 坐标影响 | 不影响地图坐标。地图位置仍以第一步选择的 POI 地址为准 |

**2.2.6 城市站类型与地址字段的关系**

| 城市站类型 | 主会场地址字段 | 地图搜索框 | 说明 |
|-----------|---------------|-----------|------|
| 线下 | 必填 | 展示 | 无地址不可提交会议 |
| 线上+线下 | 必填 | 展示 | 同上 |
| 线上 | 无此字段 | 隐藏 | 线上会议不涉及物理会场 |

**2.2.7 编辑与兼容**

| 规则项 | 内容 |
|--------|------|
| 修改地址 | 支持。办会人可清除已选地址重新搜索，或直接搜索新地址覆盖。修改第一步地址后坐标同步更新，第二步详细位置默认保留但应允许办会人重新编辑 |
| 修改详细位置 | 支持。办会人可单独修改门牌号/大厅名称等详细位置，不影响 POI 坐标和地图展示 |
| 历史数据兼容 | 已有的纯文本地址不做迁移，详情页不展示地图（保持现有纯文本展示）；仅新建或编辑后使用结构化存储的会议展示地图 |
| 地址清除 | 支持清除已选地址（点击输入框内清除按钮 ×），清空后结构化数据各字段置为 null，并同步清空详细位置字段 |

#### 验收标准

- [ ] AC-MAP-01：选择"线下"或"线上+线下"城市站类型后，主会场地址展示为两步式地址填写组件
- [ ] AC-MAP-02：选择"线上"城市站类型后，地址字段不展示
- [ ] AC-MAP-03：第一步选择地址中，输入 ≥ 2 个字符后触发 POI 搜索，300ms 防抖生效
- [ ] AC-MAP-04：搜索结果严格限定在所选城市范围内，非该城市的 POI 不展示
- [ ] AC-MAP-05：搜索结果最多展示 10 条，每条含 POI 名称（主标题）和地址（副标题）
- [ ] AC-MAP-06：无搜索结果时展示"未找到匹配的地址，请尝试其他关键词"
- [ ] AC-MAP-07：API 超时或异常时展示对应错误提示
- [ ] AC-MAP-08：选中 POI 并填写详细位置后，以结构化 JSON 存储（text / detail / display_text / lat / lng / place_id / city / district）
- [ ] AC-MAP-09：修改第一步地址后坐标同步更新
- [ ] AC-MAP-10：支持清除已选地址（清除后各字段为 null，并同步清空详细位置）
- [ ] AC-MAP-16：选中 POI 后展示详细位置输入框，支持填写门牌号、楼层、大厅名称或会议室名称
- [ ] AC-MAP-17：单独修改详细位置时，不影响地图坐标和静态地图展示

### 2.3 参会人端地图展示（MAP-02）

#### 场景描述

参会人在会议详情页底部看到当前所选城市站主会场地址对应的静态地图标注图。参会人切换城市站（如从"北京站"切换到"上海站"）时，地图相应更新。

#### 用户故事

| 编号 | 用户故事 |
|------|----------|
| US-MAP-04 | 作为参会人，我希望在会议详情页看到主会场位置的地图标注，以便直观了解会场在哪里 |
| US-MAP-05 | 作为参会人，我希望在地址文本中看到门牌号、楼层、大厅名称等详细位置，以便到达建筑后能准确找到具体会场 |
| US-MAP-06 | 作为参会人，我切换不同城市站时希望地图随之更新，以便查看各分站会场位置 |

#### 需求规格

**2.3.1 展示位置与条件**

| 规则项 | 内容 |
|--------|------|
| 展示位置 | 会议详情页底部（报名按钮上方或下方，具体位置以设计稿为准） |
| 展示条件 | 仅当当前所选城市站为"线下"或"线上+线下"类型，且主会场地址已存储结构化数据（`lat` 和 `lng` 非空）时展示 |
| 隐藏条件 | 以下任一情况不展示地图：① 所选城市站为"线上"类型；② 主会场地址未填或为纯文本（历史数据）；③ 结构化数据中坐标缺失 |
| 切换联动 | 参会人切换城市站（已有功能），地图区域根据新城市站的地址坐标重新渲染（无需页面刷新） |

**2.3.2 地图渲染**

| 规则项 | 内容 |
|--------|------|
| 地图形式 | **静态地图图片**（不可缩放/拖拽/交互），通过高德静态图 API 根据 `lat` + `lng` 生成 |
| 标注点 | 地图中央显示一个红色标注点（高德静态图 API 默认支持的 marker），标注当前会场位置 |
| 图片尺寸 | 全宽展示，高度固定（建议约 200px），适配移动端和网页端 |
| 地址文本 | 地图下方或上方展示地址的文本描述，优先取结构化数据中的 `display_text` 字段；若 `display_text` 为空，则按 `text` + `detail` 拼接展示 |
| 地图定位 | 静态地图位置仅使用第一步选中 POI 的 `lat` + `lng`，不受门牌号/大厅名称等详细位置字段影响 |
| 加载态 | 图片加载中展示骨架屏占位 |
| 加载失败 | 图片加载失败时展示降级文本（仅文字地址，不展示地图） |

**2.3.3 静态地图 URL 构造**

```
https://restapi.amap.com/v3/staticmap
  ?location={lng},{lat}
  &zoom=15
  &size={width}*{height}
  &markers=mid,,A:{lng},{lat}
  &key={高德Web服务Key}
```

参数说明：

| 参数 | 说明 |
|------|------|
| `location` | 标注点经纬度，格式 `lng,lat` |
| `zoom` | 缩放级别，建议 15（街区级，适合会场定位） |
| `size` | 图片尺寸，根据容器宽度动态计算 |
| `markers` | 标注样式：红色大号标记点 |

#### 验收标准

- [ ] AC-MAP-11：线下/线上+线下城市站已填结构化地址时，详情页底部展示静态地图 + 地址文本
- [ ] AC-MAP-12：线上城市站不展示地图区域
- [ ] AC-MAP-13：纯文本地址（历史数据）不展示地图区域，仅展示文本地址
- [ ] AC-MAP-14：切换城市站后地图随新地址坐标实时更新
- [ ] AC-MAP-15：地图图片加载失败时降级展示文本地址
- [ ] AC-MAP-18：地址文本展示包含第一步选中的 POI 地址和第二步填写的门牌号/大厅名称等详细位置
- [ ] AC-MAP-19：会议海报官网底部展示的地图位置仍以第一步选中的 POI 坐标为准，不因详细位置字段变化而偏移

## 第3章：Non-Goals（本期不包含）

| 序号 | 不包含内容 | 说明 |
|------|-----------|------|
| 1 | 交互式地图（参会人可缩放/拖拽/点击） | 静态图满足"直观展示位置"需求；交互式需加载 SDK，增加页面体积和复杂度 |
| 2 | 地图点击跳转导航（如唤起高德/百度/腾讯地图 App） | 可作为后续体验优化 |
| 3 | 多个城市站的地图同屏对比展示 | 当前为单城市站切换模式，不做多站叠加 |
| 4 | 历史纯文本地址自动迁移为结构化数据 | 变更成本高（需调用地理编码 API 逐条逆解析），且可能产生错误解析 |
| 5 | 办会人在地图上手动拖拽选点（钉选地址） | 搜索选中已覆盖核心场景，手动钉选增加实现复杂度 |
| 6 | 其他地图服务商（百度/腾讯） | 首版仅接入高德 |

## 第4章：成功指标

| 指标 | 类型 | 目标值 | 衡量方式 |
|------|------|--------|----------|
| 结构地址使用率 | 领先 | 上线 30 天内，≥80% 新建/编辑的线下会议使用结构化地址 | 结构化地址会议数 / 线下+线上+线下会议总数 |
| 地址修改率 | 领先 | 选中后修改地址的比例 ≤15% | 修改次数 / 选中次数 |
| 搜索成功率 | 领先 | 搜索后选中结果的比率 ≥90% | 选中结果次数 / 触发搜索次数 |
